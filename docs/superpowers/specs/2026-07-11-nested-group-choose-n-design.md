# Дизайн: вложенные choose-N (`selectionEntryGroup`) → лимиты в IR

**Дата:** 2026-07-11
**Статус:** утверждён к реализации
**Связано:** [[engine-parser-status]], [[real-catalogue-web-probe]], `2026-07-11-group-choose-n-design.md`, `2026-07-11-p0d-entrylink-group-resolution-design.md`

## Проблема

`group.constraint_dropped` — самый крупный «лимитный» дроп на реальном SM: **4850**, из них **4571** = choose-N на **вложенном** `<selectionEntryGroup>`. IR сегодня уплощает членов всех групп в `IrEntry.children`, но эмитит `IrGroup` (с min/max) только для групп **верхнего уровня**; вложенные под-группы прогоняются через `drop_group_constraints` (громкий дроп). Итог: лимиты вида «внутри лоадаута выбери 1 из этих» не энфорсятся — конфигурация снаряжения в билдере пропускает невалидные комбинации.

## Ключевое наблюдение (почему это только парсер)

- engine-eval энфорсит группы **плоско**: `evaluate.ts:44` идёт по `node.entry.groups`, для каждой `checkGroupConstraint` считает **прямых детей** владельца (`node.children`), чей `entry.id ∈ group.memberEntryIds`.
- `flatten_group_members` уже уплощает членов **всех уровней** (рекурсивно) в `children` владельца.
- Значит: если парсер положит вложенные группы в тот же плоский `entry.groups`, каждую со своими **прямыми** entry-членами, — enforcement заработает как есть. **engine-eval и домен-схема `IrGroup` не меняются.**

## Границы

**Делаем — только `packages/engine-parser/src/ir/map.rs`:**
- Эмитим `IrGroup` для каждой группы дерева (верхняя + вложенные) с маппящимися constraints, `member_entry_ids` = её прямые entry-члены.

**НЕ делаем:**
- Домен-схема (`@muster/domain` `IrGroup`) — без изменений (модель плоская, не дерево).
- engine-eval — без изменений (уже энфорсит плоский список).
- Условные лимиты (модификатор-на-лимите, 219) и roster-scope групповые лимиты (60) — отдельные срезы.
- Родитель-считает-вложенные (см. ограничение) — вне скоупа.

## Поведение

В `map_entry`, для каждой группы `g` в `e.groups`:
1. `flatten_group_members(g, …, &mut children)` — как сейчас (все члены всех уровней → `children`).
2. Вместо `if let Some(map_group(g)) { push }` + дропа вложенных — вызвать рекурсивный `collect_groups(g, diags, &mut groups)`:
   - `map_group(g)` (тело как сейчас, **без** внутреннего цикла `drop_group_constraints`) → если есть маппящиеся constraints, `push` в `groups`; `member_entry_ids` = `g.entries` (прямые).
   - для каждой `sub` в `g.groups` — рекурсивно `collect_groups(sub, …)`.
3. Функция `drop_group_constraints` удаляется (её роль — реальный per-constraint маппинг через существующий `map_group_constraint`, который по-прежнему громко дропает не-`selections` поле, нелокальный scope, модификатор-на-лимите).

Каждая вложенная группа честно энфорсится: её прямые члены лежат в `children` владельца, `checkGroupConstraint` их считает.

### Аддитивность и golden
Поведение групп верхнего уровня не меняется (тот же `map_group`, минус дроп-вложенных-цикл). Если в фикстуре `mini40k` нет вложенных групп — golden байт-идентичен. Проверяется в плане.

### Осознанное ограничение (never-miscompile)
Родительская группа считает только **прямые** entry-члены, а не выборы внутри вложенных под-групп. Это **ровно текущее** поведение верхнего уровня (не ухудшается): каждая под-группа при этом честно считает свои члены. Документируется комментарием; отдельной диагностики не добавляем (родительский constraint эмитится и энфорсится над своими прямыми членами — не мискомпилируется, просто не включает вложенные выборы). Точная семантика «родитель считает вложенные как единицы» — отдельный срез, если понадобится.

## Тесты

**Rust `tests/map.rs`:**
1. Entry с группой верхнего уровня, содержащей вложенную под-группу с `min`/`max` (`field=selections`, локальный scope) → `to_ir` эмитит **обе** группы в `entry.groups` с верными `memberEntryIds` и constraints; **нет** `group.constraint_dropped` для вложенной; члены обеих уплощены в `children`.
2. Вложенная группа с `roster`-scope (или не-`selections` поле) → её constraint по-прежнему `group.constraint_dropped`; сама группа не эмитится (нет маппящихся constraints).
3. Глубина ≥2 (под-под-группа) с constraint → эмитится (рекурсия доходит).

**engine-eval `test/groups.test.ts` (или evaluate):**
4. Каталог, где у entry в `groups` лежат две группы (имитируют верхнюю+вложенную), у вложенной choose-1; ростер с 2 выбранными членами вложенной → `evaluate()` даёт `group.max`. Подтверждает, что плоский список из «вложенной» группы энфорсится (кода не меняли — тест фиксирует поведение).

**Golden `tests/golden.rs`:** без изменений (mini40k без вложенных групп — проверить diff пустой).

**Осязаемая проверка (руками):** CLI на реальных .cat+.gst → `group.constraint_dropped` падает 4850→~279 (остаются 219 modifier-on-limit + 60 roster). Веб: юнит с вложенным выбором снаряжения показывает ошибку min/max при нарушении.

## Риски
- **Golden-дрейф** если mini40k всё же содержит вложенную группу с constraint → тогда это ожидаемое улучшение; перегенерировать golden и убедиться, что добавилась только вложенная группа. План проверяет diff.
- **Roster pre-seeding** (`addUnit` сидит дефолты по `entry.groups`) теперь видит и вложенные группы → их required-дефолты тоже пресидятся. Аддитивно и корректнее; roster-тесты прогоняются в общем turbo.
- **Родитель-считает-вложенные** — задокументированное ограничение, не мискомпиляция.
