# Выбор детачмента и setup-флоу армии — Design

**Дата:** 2026-07-13
**Статус:** утверждён к реализации (брейнсторм пройден)

## Контекст и проблема

Muster строит армию **matched-play** Warhammer 40k (турнирный/соревновательный
формат) на реальных каталогах BSData. Легальный matched-play список задаётся тремя
решениями уровня армии — **лимит очков → фракция → детачмент** — после чего игрок
набирает юниты. Выбранный **детачмент** (Gladius Task Force, Anvil Siege Force, …;
~10–39 на фракцию) разблокирует **только свои усиления** (enhancements — апгрейды
персонажей), усиления прочих детачментов остаются скрытыми.

Сегодня в приложении этого слоя нет вовсе: оно сразу открывает билдер на встроенной
мини-фикстуре, лимит 2000 pts захардкожен, выбора очков/фракции/детачмента нет.

Дополнительно вскрыт **баг гейтинга**, заземлённый на реальном каталоге Space Marines.
Каждая группа усилений «`<Детачмент>` Enhancements» в исходнике гейтится модификатором:

```xml
<modifier type="set" value="true" field="hidden">
  <conditionGroup type="and">
    <condition type="lessThan" value="1" field="forces"     scope="roster" childId="cac3…(Crusade Force)"/>
    <condition type="lessThan" value="1" field="selections" scope="roster" childId="d2dc…(Gladius)"/>
```

Смысл: «скрыть группу, пока (нет Crusade-форса) И (нет выбранного детачмента в
ростере)». В matched-play «нет Crusade» всегда истинно, поэтому гейт сводится к
«скрыть, пока не выбран этот детачмент».

**Корень бага — две причины, обе обязательны к устранению:**

1. **Видимость групп не смоделирована вовсе.** Гейт сидит на `selectionEntryGroup`
   (группе), а не на entry. Но `IrGroup` не имеет поля `visibilityModifiers`,
   `map_group` обрабатывает только констрейнты группы и **игнорирует её `<modifiers>`**,
   а движок/`UnitConfig` видимость групп не вычисляют. Поэтому детачмент-гейт группы
   просто теряется на входе.
2. **`field="forces"` не мапится.** Даже если бы группа несла гейт, строгий маппер
   уронил бы его из-за `forces`-условия (Crusade). Все 200 `forces`-условий каталога
   ссылаются на один `cac3` = Crusade Force.

Итог: `visibilityModifiers` у групп усилений пуст → группы видны всегда → **усиления
всех детачментов показываются одновременно**.

Важно: детачмент-часть гейта — `selections scope=roster childId=<детачмент>` —
ложится на текущий **плоский** ростер без изменений структуры дерева (force-корень
**не нужен**). Наша модель уже **уплощает** членов группы в детей владеющего entry
(`flatten_group_members`), а сами группы — лишь метаданные (члены + констрейнты) без
собственного узла в дереве вычислений. Поэтому естественное решение — **спускать
hidden-гейт группы на её entry-члены** при уплощении: члены переиспользуют уже
существующую и работающую entry-видимость (`hiddenEntryIds`), без новой машинерии
видимости групп в движке и вебе.

## Цели

1. Детачмент-гейтинг усилений **реально работает**: выбран Gladius → видны только
   усиления Gladius; сменил детачмент → набор усилений перегейчивается вживую.
2. Setup-флоу уровня армии: **лимит очков → фракция → детачмент** (Вариант A —
   гибрид: визард первого запуска, затем постоянный редактируемый setup-bar).
3. Всё встроено в текущий визуальный язык (дизайн-токены Muster), билдер не ломается.

## Не-цели (явно вне объёма)

- **Crusade / нарратив.** `field="forces"` мапится, но семантика форсов не
  реализуется — в нашей модели ростера форс-узлов нет, значит счёт форсов всегда 0
  (см. Часть 1). Этого достаточно для matched-play.
- **Реестр каталогов (библиотека всех фракций).** Грузится один каталог за раз. Шаг
  «фракция» — **заглушка на вырост**: текущая фракция активна, остальные показаны
  неактивными с плашкой «Library soon».
- **Лимит усилений ≤3/армию и «каждое ≤1».** Это отдельная задача legality-валидации;
  здесь не реализуется (усиления гейтятся по детачменту, но не считается их число).
- **Foreign-id scopes** (~90 редких условий со scope=GUID) — не трогаем; они и так
  роняют лишь свои единичные гейты, к детачмент-усилениям отношения не имеют.

## Обзор архитектуры

Срез затрагивает пять пакетов. Порядок отражает зависимости.

1. **`@muster/domain`** — добавить `"forces"` в enum `IrCondition.field`.
2. **`engine-parser` (Rust)** — (а) `map_field` мапит `"forces"` вместо дропа;
   (б) при уплощении группы её `field="hidden"` модификаторы мапятся и **спускаются
   на каждый entry-член** (`visibility_modifiers`). ~200 гейтов групп усилений так
   доезжают до членов в IR.
3. **`@muster/engine-eval`** — `aggregate` считает `field="forces"` (в нашем ростере = 0),
   и гейт члена вычисляется корректно.
4. **`@muster/roster`** — API выбора детачмента и лимита очков; helper списка детачментов.
5. **`apps/web`** — визард (Вариант A) + setup-bar + шаг выбора детачмента.

Части 1–3 чинят гейтинг (движок); Части 4–5 дают UX и проводку. После спуска гейта на
члены существующий `hiddenEntryIds` уже корректно прячет/показывает **entry-члены**
группы усилений (гейт roster-scope всегда вычисляется, в отличие от context-scope) —
то есть отображение усилений в билдере становится корректным без правок компонентов
видимости движка и веба.

## Часть 1: поле `forces` (domain + parser + engine)

### domain — `IrCondition.field`

`packages/domain/src/conditions.ts`: расширить enum поля условия:

```ts
field: z.enum(["selections", "points", "forces"]),
```

`AggregateSpec.field` в engine-eval (`scopes.ts`) — синхронно добавить `"forces"`.

### parser — два изменения в `map.rs`

**(а) `map_field` мапит `"forces"`.** Сейчас пропускает только `"selections"` и
cost-type, дающий points; всё прочее → `field_unmapped` + `None`. Добавить ветку:
`field == "forces"` → `Some("forces")`. После этого `map_condition` для forces-условия
перестаёт возвращать `None`, и строгий маппер группы (`map_condition_group_strict`)
больше не роняет модификатор из-за forces.

**(б) Спуск hidden-гейта группы на entry-члены.** `map_entry` уплощает членов группы
в детей владельца через `flatten_group_members(g, …, &mut children)`, но `<modifiers>`
самой группы нигде не обрабатываются. Изменение: при уплощении, для каждой группы `g`,
её `field="hidden"` модификаторы мапятся через `map_visibility_modifier` (тот же
строгий all-or-nothing, что и для entry-видимости — при неотображаемом условии гейт
группы дропается целиком, с диагностиком, никогда не «пере-скрывает») и **добавляются
к `visibility_modifiers` каждого уплощаемого члена этой группы**. Nested-подгруппы —
рекурсивно, гейт родительской группы также спускается на членов подгрупп (член скрыт,
если скрыт по гейту любой из своих групп-предков или по собственному eligibility-гейту).

Семантика корректна: «группа скрыта при условии C» ≡ «каждый её член скрыт при условии
C». Множественные `set hidden=true` модификаторы на члене складываются по «любой
сработал → скрыт» (уже так в движке), поэтому eligibility-гейт члена (напр. «предок —
Captain») и спущенный детачмент-гейт сосуществуют: член виден, только если он на
подходящем персонаже **и** нужный детачмент выбран.

Диагностик `condition.field_unmapped` для `forces` исчезает; на реальном каталоге
entry-члены групп усилений получают детачмент-гейт в `visibilityModifiers` (проверяемо:
у члена вроде «Adept of the Codex» появляется гейт `and(forces Crusade<1, selections
<детачмент><1)` вдобавок к eligibility-гейту).

**Golden-тест парсера байт-идентичен** — обновить эталон под новый вывод (в эталоне у
членов групп появятся спущенные `forces`-содержащие гейты). Это ожидаемое изменение
контракта.

### engine — `aggregate` для `field="forces"`

`packages/engine-eval/src/scopes.ts`, функция `aggregate`. Семантика `field="forces"`:
«сколько форсов (под-армий) заданного типа в scope». Наша модель ростера — плоский
список селектов под единственным неявным форсом; **явных форс-узлов нет**. Поэтому счёт
форсов любого конкретного `forceEntry`-id = **0**.

Реализация: при `spec.field === "forces"` вернуть `0` (в нашей модели нет узлов,
представляющих форс). Тогда для гейта усиления:

- `forces Crusade < 1` → `0 < 1` → **true** (Crusade-форса нет);
- `and`-группа сводится к `(true) AND (selections <детачмент> < 1)` = `(нет детачмента)`.

Т.е. группа скрыта ровно тогда, когда выбранный детачмент отсутствует в ростере —
искомое поведение matched-play.

> Замечание о честности модели: `forces` == 0 корректно, пока форсов в ростере нет
> (matched-play, Не-цели). Когда/если появятся явные форсы, `aggregate` для `forces`
> должен будет считать реальные форс-узлы — точка расширения помечена комментарием,
> как уже сделано для `force`/`roster` scope в `scopeNodes`.

## Часть 2: роестер — детачмент и лимит очков

### Модель хранения детачмента

Детачмент хранится как **обычный селект ростера**, а не отдельным полем. В каталоге
это корневой `selectionEntry type="upgrade" name="Detachment"` (`min=1/max=1` на выбор),
чьи дети — опции детачментов («Gladius Task Force» = entry `d2dc…`, и т.д., каждая со
своей категорией/id). Ростер содержит один селект «Detachment» с одним выбранным
ребёнком-опцией.

Плюсы такого подхода:
- Гейт `selections scope=roster childId=<детачмент>` считает этот селект **как есть** —
  никакого спец-кода в движке сверх Части 1.
- Констрейнт `min=1/max=1` на «Detachment» даёт **бесплатную legality-валидацию**:
  нет детачмента → `error` в шапке; смена — атомарна.
- Переиспользуется существующая машинерия селектов/`evaluate`.

### API `@muster/roster` (builder.ts)

```ts
// Список опций-детачментов текущего каталога (дети корневого "Detachment").
// Пусто, если каталог не содержит узла "Detachment".
export function availableDetachments(catalogue: IrCatalogue): IrEntry[];

// entryId выбранного детачмента (ребёнок-опция под корневым "Detachment"),
// либо undefined, если детачмент не выбран.
export function selectedDetachment(roster: Roster, catalogue: IrCatalogue): string | undefined;

// Установить/сменить детачмент: гарантировать ровно один селект "Detachment"
// с одним выбранным ребёнком = detachmentEntryId. Идемпотентно; смена заменяет
// ребёнка, не плодит дубли. Возвращает новый (иммутабельный) ростер.
export function setDetachment(roster: Roster, detachmentEntryId: string, catalogue: IrCatalogue): Roster;

// Изменить лимит очков армии.
export function setPointsLimit(roster: Roster, pointsLimit: number): Roster;
```

Реализация `setDetachment`: найти корневой «Detachment» entry в каталоге; если селекта
«Detachment» в ростере нет — добавить его с единственным ребёнком-опцией; если есть —
заменить набор детей на один выбранный. `availableDetachments` находит корневой
«Detachment» по имени/структуре среди `catalogue.entries` и возвращает его `children`.

`createRoster` не меняется; детачмент по умолчанию **не выбран** (визард обяжет выбрать).

### Идентификация корневого «Detachment»

Корневой узел детерминированно ищется среди `catalogue.entries` как entry с
`name === "Detachment"` и `type === "upgrade"`, несущий `min=1/max=1` констрейнт выбора.
Если каталог его не содержит (мини-фикстура), `availableDetachments` → `[]`, а web
скрывает шаг детачмента (см. Часть 3, краевые случаи).

## Часть 3: web — визард (Вариант A) + setup-bar

### Поток

**Первый запуск / новая армия:** модальный визард (переиспользует язык `.picker`,
шире — max-width ~720px) со степпером из 3 шагов:

1. **Очки** — пилюли `.uc-opt` (1000 / 1500 / 2000) + custom-инпут. → `setPointsLimit`.
2. **Фракция** — сетка карточек; текущая фракция (`catalogue.name`) активна и выбрана,
   остальные неактивны с плашкой «Library soon» + honest-note про будущий реестр.
   (Заглушка на вырост — реального выбора пока нет.)
3. **Детачмент** — двухпанельный: слева прокручиваемый список карточек
   `availableDetachments` (имя + короткий дескриптор + роль-тег; выбор = accent-обводка
   + галка), справа sticky-панель **превью усилений** под баром `.ds-section-head`.
   → `setDetachment`.

Кнопка «Start building» доступна, когда детачмент выбран (очки имеют дефолт 2000).
По завершении визард сворачивается в **setup-bar** — ряд из 3 чипов под шапкой
(Очки · Фракция · Детачмент); клик по чипу вновь открывает визард на нужном шаге.
Смена детачмента в setup-bar немедленно перегейчивает усиления в билдере (через
пересчёт `hiddenEntryIds`/`evaluate`, как уже устроено на изменение ростера).

### Превью усилений (v1, лёгкое)

Для выбранного детачмента показать список его усилений. v1-источник: группа
`selectionEntryGroup`, чьё имя = «`<имя детачмента>` Enhancements» (детерминированный
матч по имени), перечислить её дочерние entries (имя + стоимость `pointsCost`). Превью
информационное; на легальность не влияет. (Более строгий источник — группа, чей гейт
ссылается на entry-id детачмента, — возможен позже; для превью имя-матч достаточно.)

### Состояние и проводка

Состояние setup (открыт ли визард, текущий шаг) — локальный `useState` в `App.tsx`.
Признак «первый запуск» = детачмент ещё не выбран (`selectedDetachment` === undefined)
→ визард открыт автоматически. Points-лимит из ростера (`roster.pointsLimit`), фракция —
`catalogue.name`, детачмент — `selectedDetachment`.

Существующий билдер (двухколоночный ростер+датащит, `AddUnitPicker`, `UnitConfig`,
фильтрация опций через `hiddenEntryIds`) **не меняется** — визард и setup-bar живут
над ним. Гейтинг усилений в `UnitConfig` начинает работать сам после Частей 1–3.

### Мобилка (< 640px)

Двухпанельный шаг детачмента складывается в один столбец (список над превью); setup-bar
переносит чипы; визард — `max-height: 92vh`. Всё уже предусмотрено в макете.

## Поток данных: «выбрал детачмент → усиления перегейтились»

1. Пользователь выбирает Gladius в визарде/setup-bar → `setDetachment(roster, d2dc, catalogue)`.
2. Ростер получает селект «Detachment» с ребёнком Gladius (entryId `d2dc`).
3. `evaluate`/`hiddenEntryIds` строят state; roster-scope `aggregate(selections, d2dc)` = 1.
4. Спущенный на членов гейт «Gladius Enhancements»: `(forces Crusade<1 = true) AND
   (selections d2dc<1 = false)` → **false** → члены **не** скрыты → усиления Gladius
   доступны в `UnitConfig` (через `hiddenEntryIds`).
5. Гейт членов «Anvil Enhancements»: `(true) AND (selections <Anvil> < 1 = true)` →
   **true** → члены скрыты → усиления Anvil не показываются. ✓
6. Смена детачмента на Anvil инвертирует картину вживую.

## Обработка ошибок и краевые случаи

- **Детачмент не выбран:** все группы усилений скрыты (гейт `нет детачмента` = true) —
   корректно. Плюс `min=1` констрейнт даёт `error` «нужен детачмент» в шапке.
- **Каталог без узла «Detachment»** (мини-фикстура): `availableDetachments` → `[]`;
   web пропускает шаг детачмента в визарде и не рисует чип детачмента в setup-bar.
   Приложение остаётся рабочим на фикстуре (регресс-безопасность существующих тестов).
- **Смена детачмента при уже добавленных усилениях:** усиления другого детачмента
   становятся скрытыми, но остаются в данных и **стоят очков** — существующий сигнал
   `hiddenSelectionIds` уже помечает такие узлы как «стали недоступны» (warning, не
   удаление). Отдельной работы не требуется.
- **Custom-очки:** валидировать > 0; пустой/невалидный ввод не меняет лимит.

## Стратегия тестирования

- **domain:** unit — `IrCondition` парсит `field: "forces"`; round-trip.
- **parser (Rust):** unit — `map_field("forces")` → `Some("forces")`; условие с
   `field="forces"` больше не роняет модификатор. Отдельный unit — **спуск hidden-гейта
   группы на членов**: `selectionEntryGroup` с `field="hidden"` модификатором →
   каждый уплощённый член получает этот гейт в `visibility_modifiers` (плюс свой
   eligibility-гейт, если был); nested-подгруппы наследуют гейт предка; неотображаемое
   условие роняет гейт группы целиком (диагностик, никогда не пере-скрывает). Обновить
   **golden** эталон (байт-идентичность) под новый вывод.
- **engine-eval:** unit — `aggregate` при `field="forces"` = 0; сквозной тест гейта:
   маленький каталог (детачмент-entry + группа усилений с гейтом
   `and(forces X<1, selections DET<1)`) → без детачмента группа скрыта, с детачментом
   в ростере — видна; смена детачмента инвертирует. Покрытие 100% (планка пакета).
- **roster:** unit — `availableDetachments`, `selectedDetachment`, `setDetachment`
   (добавление, замена без дублей, идемпотентность), `setPointsLimit`. Планка 100%.
- **web:** component (jsdom) — визард проходит очки→фракция→детачмент; выбор детачмента
   зовёт `setDetachment`; setup-bar открывает визард на шаге; шаг детачмента скрыт при
   пустом `availableDetachments`. Существующие тесты билдера остаются зелёными.
- **сквозная проверка на реальном SM:** загрузить реальный каталог, выбрать Gladius →
   в `UnitConfig` персонажа видны только Gladius-усиления; сменить на Anvil → набор
   меняется. Проверка в браузере (preview) со скриншотом.

## Влияние на legality-валидацию (matched-play)

- «Ровно один детачмент» — обеспечивается существующим `min=1/max=1` констрейнтом на
   «Detachment» (появляется как `error` в шапке при 0/2 детачментах).
- «≤3 усиления, каждое ≤1» — **вне объёма** этого среза (отдельная задача); детачмент-
   гейтинг усилений от неё независим.
