## Как это устроено в New Recruit / BattleScribe

Настройка — это **дерево выбора**, где тип контрола для каждого узла **выводится из его ограничений** (`constraint field=selections`, `type min/max`, `scope`), а не задаётся вручную.

**Группа (`selectionEntryGroup`)** ограничивает, сколько её членов выбрать:
- `max=1, min=1` → **радио обязательное** (всегда одно; смена, не обнуление).
- `max=1, min=0` → **радио опциональное** (одно или «ничего»).
- `max=N (>1)` → **до N** (чекбоксы/тумблеры членов, блок сверх лимита).

**Опция (`selectionEntry`)** ограничивает собственный счётчик (свои `constraint scope=self|parent`):
- `max=1` → **тумблер** (0/1).
- `max=N (>1)` → **степпер −/+** в границах `min..max` (напр. 5–10 моделей).
- `min=max=k` → **фиксировано**, не редактируется (базовое снаряжение).

**Defaults:** при добавлении юнита группы авто-выбирают `defaultSelectionEntryId`, опции авто-заполняются до `min`. Юнит появляется с базовым лоадаутом, дальше правится.

**Валидация** min/max — уже делает наш `evaluate()`.

## Что уже есть vs чего нет

- `IrConstraint` уже несёт `type/value/field(selections|points)/scope(self|parent|force|roster)/targetType/targetId/includeChildSelections` — данных для «умных контролов» **достаточно**.
- `IrGroup.constraints` (min/max, selections) — есть; групповая замена/лимит (`toggleGroupMember`) — **сделано** (commit `34cb7b5`).
- **Нет:** различения обязательного/опционального радио в UI; степпера в границах опции; фиксированных; и `defaultSelectionEntryId` (парсер его не эмитит).

## Scope — два захода

### Заход 1 — умные контролы из min/max (без изменений парсера)

**`@muster/roster`** — новая чистая функция-классификатор (данные уже в IR):
```ts
// Как редактировать эту группу под юнитом.
type GroupControl =
  | { kind: "single"; required: boolean }   // max===1; required = (min>=1)
  | { kind: "multi"; max: number };          // max>1 (Infinity если max нет)
groupControl(group: IrGroup): GroupControl

// Как редактировать эту опцию (её собственный счётчик).
type OptionControl =
  | { kind: "toggle" }                        // max===1
  | { kind: "stepper"; min: number; max: number } // max>1
  | { kind: "fixed"; count: number };         // min===max (>0)
optionControl(entry: IrEntry): OptionControl
```
Правила выбора max/min берём из `entry.constraints` с `field==="selections"` и `scope∈{self,parent}` (own-count), `type` min/max; `max` отсутствует → `Infinity`, `min` отсутствует → `0`. Групповые — из `group.constraints`.

Плюс расширить поведение выбора под контрол:
- `setCount` уже есть — UI ограничивает ввод диапазоном `min..max` (клампит).
- Для обязательного `single`-радио (`required`) — запретить обнуление: `toggleGroupMember` по уже выбранному члену обязательной группы = **no-op** (нельзя оставить группу пустой; можно только сменить на другого). Добавить параметр/ветку, НЕ ломая опциональный случай. Реализация: если член уже выбран И это единственный выбранный И `required` — вернуть роняем без изменений. (Передаём `required` в `toggleGroupMember`, либо новая тонкая обёртка — решить в плане.)

**`apps/web` `UnitConfig`** — рендерить по классификатору:
- группа `single` → набор радио-кнопок членов (обязательное: без «снять»; опциональное: клик по выбранному снимает). Текущая toggle-логика уже близка.
- группа `multi` → тумблеры членов до `max` (как сейчас).
- опция вне групп с `stepper` → **−/+ степпер** в границах, вместо сырого `count`-инпута.
- опция `toggle` → не показывать бессмысленный `count`.
- `fixed` → показать «×k (fixed)», без контролов.

Точка оценки: на mini40k Wargear ведёт себя как радио; если у Battle Squad есть счётный дочерний — как степпер. Пощупать сразу.

### Заход 2 — defaults + проход парсера

**Парсер (Rust):** читать `defaultSelectionEntryId` у `selectionEntryGroup` (и, если встретится, у entry), протаскивать через resolve и эмитить в IR. Golden mini40k может остаться байт-идентичным, если у него нет defaults — тогда добавить дефолт в фикстуру ОТДЕЛЬНО и с новым golden-снимком (в плане).

**Домен:** `IrGroup` получает `defaultMemberEntryId?: string`.

**`@muster/roster`:** `addUnit`/`addOption` авто-populate поддерева при добавлении:
- для каждой группы юнита с `defaultMemberEntryId` или `min>=1` — добавить дефолт (или первого члена) в нужном числе;
- для каждой дочерней опции с `min>=1` — включить со счётом `min`;
- рекурсивно по поддереву. Идемпотентно и иммутабельно.

**`apps/web`:** без изменений логики — юнит просто появляется преднаполненным.

Точка оценки: «Add Captain» → приходит с дефолтным оружием и очками базового лоадаута, как в NR.

## Вне scope (осознанно)

- Условная видимость (`hidden` + `condition`) = детачмент-гейтинг — это **S2**, отдельно.
- Реальные GW-каталоги в git не идут; проверка Захода 2 на реальном .cat — локально, вне git, инспектором.
