# S1: хребет ростера + тонкий веб-билдер — дизайн

**Дата:** 2026-07-11
**Статус:** утверждён (brainstorming), готов к плану
**Контекст:** первый под-проект «ростер-билдер как New Recruit / War Organ» (см. развилку в диалоге). Дальше: S2 — условная доступность (детачмент-гейтинг усилений), S3 — мобайл.

## Проблема и цель

Парсер даёт богатый каталог-IR (юниты, оружие «выбери-N», очки, категории, форс-орг), а `engine-eval` уже **чисто** считает очки и валидацию (`evaluate(roster, catalogue): ValidationResult`). Чего нет: **слоя сборки ростера** (добавить юнит, выбрать опцию, «что доступно») и **интерактивного UI**, чтобы это можно было собрать и пощупать.

Цель S1: пользователь **собирает ростер в браузере** — добавляет юниты, настраивает оружие, видит живые очки и валидацию. Хребет, на который встанут S2 (гейтинг) и S3 (мобайл).

## Что уже есть (не трогаем по сути)

- `@muster/domain`: `Roster { pointsLimit, selections: RosterSelection[] }`, `RosterSelection { id, entryId, count, selections[] }`, `IrCatalogue`, `ValidationResult { valid, totalPoints, pointsLimit, issues, dismissed, hasHouseRules }`.
- `@muster/engine-eval`: `evaluate(roster, catalogue)` — чистый расчёт очков + все ограничения + overrides. Переиспользуем как есть.

## Scope

**В scope:**
- Новый пакет `@muster/roster` — чистый слой редактирования/доступности поверх домена (без UI).
- Новое приложение `apps/web` — тонкий React-билдер, крутящий **настоящий** `@muster/roster` + `engine-eval`.
- Каталог: встроенная `mini40k` IR-фикстура (в репозитории) как дефолт; загрузка своего IR-JSON (для реальных фракций — GW-данные в git не идут).

**Вне scope (осознанно):**
- Детачмент-гейтинг / условная доступность — **S2**.
- Армейские правила усилений (≤3/армию, все разные) — **S2**.
- Персистентность, синк, экспорт, аккаунты — позже.
- Мобайл (Expo) — **S3** (слой `@muster/roster` уже кросс-платформенный, переиспользуется).
- Профили/статлайны, стратагемы — вне (стратагем нет в данных).

## Архитектура

Слои снизу вверх: `domain` (типы) → `engine-eval` (чистый расчёт, есть) → **`@muster/roster`** (чистое редактирование, новое) → **`apps/web`** (UI, новое). UI держит один `Roster` в состоянии, на каждую мутацию зовёт функции `@muster/roster`, затем `evaluate()` для отображения.

### Пакет `@muster/roster` (чистый TS)

Зависит от `@muster/domain`. Все функции **иммутабельны** (возвращают новый `Roster`); id селекций генерятся детерминированно-уникально.

```ts
// Создать пустой ростер под каталог.
createRoster(catalogue: IrCatalogue, pointsLimit: number, name?: string): Roster

// Какие юниты можно добавить в корень (корневые IrEntry каталога).
availableUnits(catalogue: IrCatalogue): IrEntry[]

// Добавить юнит (корневую селекцию) / вложенную опцию.
addUnit(roster: Roster, entryId: string): Roster
addOption(roster: Roster, parentSelectionId: string, entryId: string): Roster

// Изменить число моделей / убрать селекцию (и её поддерево).
setCount(roster: Roster, selectionId: string, count: number): Roster
remove(roster: Roster, selectionId: string): Roster

// Что выбирается под данной селекцией: её опции-дети и группы «выбери-N».
// Питает меню UI. Возвращает сами IrEntry/IrGroup из каталога.
optionsFor(roster: Roster, selectionId: string, catalogue: IrCatalogue):
  { options: IrEntry[]; groups: IrGroup[] }
```

Модель «что доступно» для тонкой версии: опции юнита = `children` его `IrEntry` (кандидаты) + члены его `groups`; пользователь включает нужное (каждое → дочерняя `RosterSelection`). Точную «обязательность» (min=0 vs required) не вычисляем — показываем как переключаемые, а корректность (`group.min/max`, `constraint`) досчитывает `evaluate()`. Это честно и тонко.

Тесты: юнит-тесты на каждую функцию (иммутабельность, добавление/удаление поддерева, генерация id, availableUnits/optionsFor против фикстуры). Держим планку строгости пакетов ядра.

### Приложение `apps/web` (Vite + React + TS)

Зависит от `@muster/roster`, `@muster/engine-eval`, `@muster/domain` (workspace). Один экран, три зоны:

- **Палитра юнитов** (слева): `availableUnits(catalogue)` — карточки с именем и очками; клик → `addUnit`.
- **Ростер** (справа): полоса очков `totalPoints / pointsLimit` (из `evaluate`), список добавленных юнитов, живой список `issues` (ошибки/предупреждения). Клик по юниту → раскрыть настройку.
- **Настройка юнита**: `optionsFor(...)` — группы «выбери-N» как переключатели (choose-1 → радио, иначе чекбоксы), число моделей (`setCount`), удалить (`remove`).

Поток данных: состояние = `{ roster }`; любая мутация → новая `roster` → `evaluate(roster, catalogue)` → перерисовка очков/ошибок. Детачмент выбирается как обычный юнит/опция (без гейтинга — S2).

**Источник каталога:** встроенный `mini40k.ir.json` (копия golden-фикстуры парсера — маленькая, валидная, без GW-IP) как дефолт; кнопка «загрузить IR-JSON» валидирует через `IrCatalogue` (Zod) и заменяет каталог — так грузятся реальные фракции локально, вне git.

Тесты: тонкий шелл — один smoke/компонентный тест (рендер + добавление юнита меняет очки). Полное покрытие не требуем (UI).

## Точка оценки заказчика

Запущенный билдер (`apps/web` через dev-сервер): реально собираешь ростер на mini40k — добавляешь юниты, крутишь оружие, видишь очки и валидацию вживую. Даю превью/скриншот. (Это твоя «щупательная» точка вместо ревью спеки.)

## Definition of Done

- `@muster/roster`: функции сборки/доступности, иммутабельные, покрыты тестами.
- `apps/web`: тонкий билдер — добавить юнит, настроить оружие («выбери-N» переключателями), число моделей, удалить; живые очки (`totalPoints / pointsLimit`) и список валидации из настоящего `evaluate()`; на встроенном mini40k; загрузка своего IR работает.
- `domain` и `engine-eval` не тронуты (только зависим от них).
- Собирается и запускается (`pnpm --filter web dev`); `turbo` build/test проходят.
