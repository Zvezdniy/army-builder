# Дизайн + план: цена реальных юнитов по имени стоимости "pts"

**Дата:** 2026-07-12
**Статус:** утверждён к реализации (blanket-делегирование)
**Связано:** [[roster-web-status]]

## Проблема (корректность — билдер показывает 0 очков)

`nodePoints`/`effectiveNodePoints` находят стоимость по `c.name === "points"`. Реальные каталоги BSData Warhammer 40k хранят авторитетную стоимость под именем **`"pts"`** (напр. `pts=80`), а `"points"` присутствует со значением **0**. → каждый реальный юнит оценивался в **0 очков**, что делает билдер бесполезным. (Mini-фикстуры используют `"points"`, поэтому баг не всплывал.)

## Решение

Общий хелпер `pointsCost(entry): IrCost | undefined` в `cost.ts`, матчящий имена из списка `["pts", "points"]` в порядке приоритета (сначала `"pts"`, потом `"points"`) — потому что реальные каталоги несут ОБА (`pts`=реальное, `points`=0); матч `"points"` первым дал бы 0. Оба потребителя используют хелпер:
- `cost.ts` `nodePoints` — `pointsCost(node.entry)`;
- `resolve.ts` `effectiveNodePoints` — `pointsCost(node.entry)` (модификаторы стоимости применяются как прежде).

## Инвариант «не сломать существующее»
- Mini-фикстуры (`"points"`) продолжают матчиться (fallback) → все существующие cost-тесты зелёные.
- Реальные каталоги (`"pts"`) теперь оцениваются корректно.
- Хелпер DRY (одно место знания имён), используется обоими путями стоимости — база и модифицированная согласованы.
- Никаких изменений схемы/парсера; только engine-eval.

## Тесты (engine-eval, 100%)
- `pointsCost`: матчит `"pts"`; матчит `"points"`; при обоих — предпочитает `"pts"` (не 0); ни одного → undefined.
- `nodePoints`: `pts` × effectiveCount; отсутствие → 0.
- Существующие `totalCost`/resolve-тесты (на `"points"`) зелёные.

## Осязаемо
Реальный SM: Intercessor Squad 160, Aggressor 190, Hellblaster 220, Eradicator 180 очков (было 0). Проверено через `evaluate()` на реальном каталоге.

## Известное ограничение (вне этого среза)
Модель-count-условное ценообразование (breakpoints 5/10 моделей через условные cost-модификаторы) — отдельная корректность; здесь фиксируется только выбор правильного имени стоимости, чтобы реальные очки вообще считались.
