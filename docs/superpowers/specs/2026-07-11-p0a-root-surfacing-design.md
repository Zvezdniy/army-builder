# P0-a: всплытие корней каталога (top-level entryLinks) — Дизайн

**Дата:** 2026-07-11
**Статус:** Одобрено, готово к планированию
**Область:** Muster (engine-parser). Домен и engine-eval — без изменений.

## Цель

Научить парсер видеть **выбираемые корни** реального BattleScribe-каталога и
эмитить их в IR. Сейчас корни ростера — это top-level `<entryLinks>` каталога
(в `Imperium - Space Marines.cat` их 161), указывающие в `<sharedSelectionEntries>`.
Парсер эти каталог-левел линки **вообще не читает** — они молча выпадают на этапе
сырого разбора, поэтому IR любой реальной фракции получается пустым
(`entries: []`) при нулевой диагностике (см. отчёт разведки
`2026-07-11-real-catalogue-recon.md`). Это первый срез из двух: **P0-a** делает IR
непустым; **P0-b** (отдельная спека) добавит межфайловую сборку `.gst`+`.cat`,
чтобы категории и очки резолвились и заработала форс-орг-валидация.

## Рамки

**В рамках**
- Чтение каталог-левел `<entryLinks>` в сырой модели.
- Разворачивание (инлайнинг) каждого разрешимого корневого линка в корневой
  `IrEntry`, переиспользуя существующую машинерию инлайнинга entryLinks — с теми
  же cycle-guard и node/depth-капами (общий бюджет, не пер-корень).
- Громкая диагностика + drop для **любого** entryLink (корневого ИЛИ вложенного),
  чью цель нельзя разрешить в этом файле — никогда не выдуманное поддерево.
  **Критично:** сейчас вложенная неразрешимая ссылка роняет весь резолв жёсткой
  ошибкой `UnresolvedRef`. В реальном SM-файле 28% всех entryLink (662 из 2306)
  указывают в `.gst`/библиотеки; без этой толерантности разворачивание корней
  просто уронит парс целиком (сегодня файл «парсится» лишь потому, что корни не
  читаются и эти поддеревья не обходятся). Поэтому `resolve` меняет поведение на
  «диагностика + пропустить ребёнка» и получает канал диагностик.

**Явно вне рамок** (уедет в P0-b / P1; в этом срезе — сохраняется как opaque id
или тихо не трогается, как сегодня)
- **Фактический резолв** cross-file ссылок (`catalogueLink` — в SM 4 шт. — и всех
  висячих entryLink) — **P0-b**. В P0-a они лишь диагностируются и дропаются, а не
  разрешаются.
- Категории (899/1111 `categoryLink` указывают в `.gst`) и costType (9/11 в
  `.gst`/библиотеках): их **id уже сохраняются в IR как opaque-строки** (категории
  в IR — просто список id; неизвестный costType → пустое имя, известный минорный
  случай). Определения (имена категорий, force-констрейнты) придут в **P0-b**.
- `profiles`/`characteristics` (статлайны), `modifierGroup`, `infoLinks`, `rules` —
  **P1**, не трогаем.

## Ключевое решение: где разворачивать корни

Корневой entryLink каталога семантически идентичен entryLink **внутри** сущности:
«вставь сюда резолвнутую копию цели». Существующий `resolve_entry`
(`resolve/links.rs`) уже делает ровно это для вложенных линков — с path-visited
cycle-guard и общими `MAX_RESOLVED_NODES`/`MAX_RESOLVE_DEPTH` капами. Поэтому
разворачивание корней делаем **в `resolve`**, переиспользуя `resolve_entry`, а не в
маппинге: только у `resolve` есть символьная таблица, инлайнинг и, критично,
DoS-капы (130 корней тянут поддеревья — бюджет узлов должен быть **общим**, иначе
пер-корневой обход обойдёт защиту).

`resolve` при этом получает канал диагностик (`&mut Vec<Diagnostic>`): раз резолв
теперь **терпимо** относится к неразрешимым ссылкам (диагностика + drop вместо
жёсткой ошибки), он обязан эти события фиксировать. Это единая точка эмиссии для
всех `entryLink.unresolved` — и корневых, и вложенных, поэтому маппинг вообще не
трогаем.

Домен IR **не меняется**: корень — обычный `IrEntry`. Никакого нового доменного
типа, никаких правок engine-eval, никаких изменений Zod-схемы или golden mini40k
(там нет каталог-левел линков и все вложенные ссылки резолвятся в файле) —
кросс-языковой контракт остаётся цел, blast radius минимален.

## Слой 1 — сырой разбор (`raw`)

- `RawCatalogue` получает поле:
  ```rust
  pub entry_links: Vec<RawEntryLink>,   // catalogue-level <entryLinks> (roots)
  ```
- В `read_catalogue` (`raw/parse.rs`) добавить match-арм на прямой `<entryLinks>`
  каталога (сейчас его нет — линки падают в catch-all skip):
  ```rust
  b"entryLinks" => read_entrylinks_into(&mut cat.entry_links, &mut r)?,
  ```
  Функция `read_entrylinks_into` уже существует (используется для entryLinks
  внутри сущностей и групп).

## Слой 2 — резолв (`resolve`)

`resolve`/`resolve_with_caps`/`resolve_entry`/`resolve_group` получают параметр
`diags: &mut Vec<Diagnostic>`. Два изменения поведения:

**(a) Вложенные ссылки — толерантность вместо жёсткой ошибки.** В `resolve_entry` и
`resolve_group` заменить
```rust
let target = symbols.entry(&link.target_id).ok_or_else(|| ParseError::UnresolvedRef(...))?;
```
на: если цель найдена — как сейчас (инлайн); если **не найдена** — выдать
диагностику `entryLink.unresolved` и **пропустить** этого ребёнка (`continue`), не
роняя резолв. `UnresolvedRef` для entryLink больше не производится (вариант ошибки
оставить в enum — он безвреден; ни один тест на него не завязан). Cycle-guard,
node/depth-бюджет — **без изменений** (это по-прежнему жёсткие `ParseError`).

**(b) Разворачивание корней.** В `resolve_with_caps`, после резолва `cat.entries`,
пройти по `cat.entry_links`:
- цель найдена → `resolve_entry(target, ...)` тем же общим бюджетом/капами и свежей
  проверкой цикла (как в существующем цикле по ссылкам, строки links.rs:60–65) →
  результат **добавить в `cat.entries`**;
- цель не найдена → диагностика `entryLink.unresolved`, drop.
В конце `cat.entry_links` очистить. Разрешённые корни лежат в `cat.entries` в
конце (после исходных top-level `<selectionEntries>` — для mini40k их нет, порядок
не меняется).

Единый код диагностики для обоих случаев: `entryLink.unresolved`, сообщение
`format!("entryLink target {} not found in this file (dropped)", link.target_id)`.

## Слой 3 — маппинг (`ir/map.rs`) + связка (`lib.rs`)

- **Маппинг — без изменений.** Разрешённые корни уже в `cat.entries` и текут через
  `map_entry` как обычные сущности; всю диагностику по неразрешимым ссылкам выдаёт
  `resolve`.
- **`parse_bytes` (`lib.rs`) — протянуть диагностики резолва.** Сейчас
  `let resolved = resolve(raw)?; Ok(to_ir(&resolved))`. Станет: завести
  `let mut diags = Vec::new();`, вызвать `resolve(raw, &mut diags)?`, затем
  `let (ir, map_diags) = to_ir(&resolved); diags.extend(map_diags); Ok((ir, diags))`.
  Так наружу (`parse_bytes`/`parse_file`/CLI) уходят и диагностики резолва, и
  маппинга.

## Кросс-языковой контракт и фикстуры

- **mini40k golden не меняется** (нет каталог-левел entryLinks) → golden парсера и
  копия в engine-eval остаются байт-идентичными; engine-eval не трогаем.
- **Новый парсер-тест** (синтетический inline-XML в `tests/map.rs` и/или
  `tests/resolve.rs`), проверяющий:
  1. Каталог с 2 top-level `<entryLinks>` на shared-сущности → 2 корневых
     `IrEntry` всплывают, с инлайненными поддеревьями (напр. вложенный upgrade).
  2. Shared-сущность, на которую **нет** корневого линка, в `entries` **не
     попадает** (всплывают только слинкованные корни, не весь shared-пул).
  3. Висячий top-level entryLink (цель не в файле) → диагностика
     `entryLink.unresolved` + корень не эмитится.
  4. **Вложенная** висячая ссылка (корень резолвится, но его ребёнок-entryLink
     указывает на отсутствующую цель) → резолв **не падает**: диагностика
     `entryLink.unresolved`, ребёнок пропущен, сам корень и остальные его дети
     эмитятся. (Ключевой сценарий — именно он делает реальный файл парсимым.)
  5. Безопасность: top-level entryLink в цепочку со ссылкой-циклом → типизированный
     cycle-error (`ParseError`), как для вложенных линков; общий node-бюджет
     действует.
- **Проверка на живом файле — ручной шаг в scratchpad, не коммит-тест** (сам
  `.cat` в git не кладём: GW-IP-adjacent, 3.6 MB). Ожидаемо: `Imperium - Space
  Marines.cat` → ~130 корневых `IrEntry` (было 0); множество диагностик
  `entryLink.unresolved` (порядок ~690: ~31 корневой + ~662 вложенных cross-file);
  один реальный юнит, вручную собранный в ростер, оценивается движком (сумма очков
  верна по локальным costType; категории сохранены как id).

## Глобальные ограничения (обязательны для каждой задачи)

- **Никогда не считать неправильно / громкая неполнота.** Любая неразрешимая
  ссылка (корневая или вложенная) — диагностика `entryLink.unresolved` + drop,
  никогда не выдуманное поддерево. То, что вне среза, — видно в диагностиках, а не
  молчит и не роняет парс.
- **Недоступный вход.** Разворачивание корней и толерантный резолв используют общий
  node/depth-бюджет и path cycle-guard (`MAX_RESOLVED_NODES`, `MAX_RESOLVE_DEPTH`) —
  без изменений; крейт остаётся `#![forbid(unsafe_code)]`.
- **Контракт синхронен.** mini40k golden парсера и копия в engine-eval — байт-в-байт
  идентичны и не меняются; golden-тест и кросс-языковой контракт-тест зелёные.
- **Гейты.** `cargo test`, `cargo clippy --all-targets -- -D warnings
  -A clippy::single_match -A clippy::while_let_loop`, `cargo deny check`,
  `cargo audit` — чисто. `pnpm test`/`pnpm typecheck` не затронуты, остаются
  зелёными.
