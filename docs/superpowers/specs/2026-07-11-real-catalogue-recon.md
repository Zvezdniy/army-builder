# Разведка ingestion реальных каталогов — отчёт о пробелах

**Дата:** 2026-07-11
**Тип:** investigation-спайк (код парсера не менялся)
**Вход:** BSData `wh40k-10e` — `Warhammer 40,000.gst` (705 KB) + `Imperium - Space Marines.cat` (3.6 MB), скачаны в scratchpad (не в git; GW-IP-adjacent, MIT-данные).
**Цель:** прогнать реальные данные текущим парсером, собрать фактический список пробелов, задать приоритет настоящей спеки ingestion по фактам.

## Метод

Двусторонний, потому что парсер **молча пропускает** нераспознанные элементы
(`skip_element`, без диагностики) — «нет диагностики» ≠ «представлено».

1. **Динамика** — `parse_file` (release CLI `muster-parse`) по `.gst` и `.cat`;
   собраны exit-код, диагностики по кодам, размер IR.
2. **Статика** — инвентаризация XML-тегов и их частот в исходниках, сопоставление
   с тем, что реально моделирует `raw/parse.rs` + `ir/map.rs`.
3. **Структура корней** — разбор прямых детей `<catalogue>`/`<gameSystem>`, чтобы
   понять, откуда берутся выбираемые юниты.

## Headline-находка

| Файл | Размер входа | exit | Размер IR | Диагностик |
|---|---|---|---|---|
| `Warhammer 40,000.gst` | 705 KB | 0 (успех) | **695 байт**, `entries: []` | 1 |
| `Imperium - Space Marines.cat` | 3.6 MB | 0 (успех) | **172 байта**, `entries: []` | **0** |

**Парсер «успешно» проглотил всю армию, вернув пустой IR и ноль диагностик.**
3.6 MB реальных правил (1102 `selectionEntry`, 389 групп, 2691 constraint, 2560
modifier, 1193 профиля) превратились в 172 байта с `entries: []` — без единого
предупреждения. Это самый опасный класс поведения для проекта, чей принцип №1 —
«никогда не считать неправильно»: здесь мы не мискомпилируем, но **молча теряем
всё**, и наружу это выглядит как валидный успех.

## Корневая причина: корни ростера не эмитятся

Прямые дети `<catalogue>` в Space Marines:

| Контейнер | Кол-во | Что это |
|---|---|---|
| `categoryEntries` | 143 | определения категорий |
| **`entryLinks`** | **161** | **корни ростера** — выбираемые юниты (ссылки в shared) |
| `sharedSelectionEntries` | 276 | определения юнитов (цели ссылок) |
| `sharedSelectionEntryGroups` | 33 | общие группы |
| `sharedProfiles` | 11 | общие профили |
| `catalogueLinks` | 4 | ссылки на другие каталоги/`.gst` |
| `profileTypes`, `costTypes`, `publications`, `infoLinks` | 1/2/2/2 | типы и мета |

Реальный каталог держит юниты в `sharedSelectionEntries`, а **выбираемые корни —
это top-level `<entryLinks>` (161 шт.)**, указывающие в shared. Парсер индексирует
`sharedSelectionEntries` в символьную таблицу, инлайнит entryLinks *внутри*
сущностей, но **не эмитит top-level `<entryLinks>` каталога как корневые
`IrEntry`**. Мэппер отдаёт только прямые `<selectionEntries>` каталога — а их на
верхнем уровне нет. Итог: `entries: []`. Аналогично `.gst`: юниты/силы туда не
кладут, `forceEntries` (4) и `categoryEntries` (114) есть, но выбираемых
сущностей нет — они в `.cat`.

## Пробелы по приоритету

### P0 — без этого реальные данные не доходят до движка вообще

- **Top-level `<entryLinks>` каталога не становятся корнями IR.** Это гейт: пока
  161 корневой линк не разворачивается в `IrEntry` (через уже существующий
  инлайнинг shared-целей), IR любой реальной фракции пуст. Ссылка: `ir/map.rs`
  эмитит только `catalogue.entries`; нужен путь «catalogue top-level entryLinks →
  resolve → IrEntry».
- **Межфайловая сборка `.gst` + `.cat` (+ `catalogueLink`).** Категории,
  costTypes, force-энтри, часть shared-сущностей и профили-типы определены в
  `.gst`; `.cat` содержит `catalogueLinks` (4) на них/другие каталоги. Одиночный
  парс каждого файла структурно неполон — символьную таблицу нужно строить из
  всего набора файлов системы. Сейчас `parse_file` берёт ровно один файл.

### P1 — тихая потеря правил внутри визуально «распарсенного» юнита

- **`modifierGroup` / `modifierGroups` (214 / 211) не распознаются.** Их нет в
  списке тегов `raw/parse.rs` → **тихо пропускаются вместе со всеми вложенными
  модификаторами**. Это прямой удар по «никогда не считать неправильно»: цена/
  лимиты, изменяемые сгруппированными условными модификаторами, молча теряются →
  неверные очки/валидация без диагностики. Кандидат №1 после P0.
- **Профили и характеристики (`profiles` 1193, `characteristics` 4437) не
  моделируются.** Это статлайны (Move/Toughness/Wounds, статы оружия) — то, что
  игрок реально видит. Тихо пропускаются. Для **валидации и очков не нужны**, но
  без них билдер бесполезен для человека. Требует нового понятия в `@muster/domain`
  (профиль/характеристика) — существенное расширение IR, отдельным треком.

### P2 — display-only, на валидацию/очки не влияет

- `infoLink`/`infoLinks` (923 / 436) — ссылки на общие профили/правила.
- `rule`/`rules` (40/39), `description`, `publication` (2), `comment` (312) —
  текст правил и флавор.
- `association`/`associations` (3) — новая механика BS (напр. «прикреплённый
  лидер»); редка, но появляется.

## Мета-находка (гигиена, важнее, чем кажется)

Парсер молча скипает неизвестные элементы **без диагностики**. Для проекта с
инвариантом «никогда не считать неправильно» это дыра: «молча неполно» так же
опасно, как «неправильно», и сейчас невидимо. Рекомендация к спеке: при пропуске
значимого элемента (например `modifierGroup`, `profile`, нераспознанный корневой
контейнер) эмитить диагностику `*.unrepresented`, чтобы неполнота была громкой, а
не тихой. Тогда любой будущий прогон реальных данных сам сообщит, что потерял.

## Выданные диагностики (факт)

- `.gst`: 1 × `constraint.force_global_unrepresentable` (force-констрейнт без
  categoryLink — уже известный представимый случай).
- `.cat`: 0 — потому что поддеревья юнитов вообще не посещаются (корней нет).

Отсутствие диагностик здесь — не признак здоровья, а следствие того, что обход не
дошёл до контента.

## Safety-капы: блокеров нет

`MAX_INPUT_BYTES` = 64 MB (Space Marines 3.6 MB влезает свободно); `MAX_XML_NODES`
и `MAX_RESOLVED_NODES` = 5M. На одиночном файле упора в капы не было. При
межфайловой сборке всей системы это стоит перепроверить (совокупный размер и
fan-out резолва), но для одной фракции + `.gst` запас большой.

## Рекомендованная последовательность настоящей спеки ingestion

1. **P0-a: корни из top-level `<entryLinks>` каталога** — самый дешёвый способ
   получить непустой IR реальной фракции; переиспользует существующий инлайнинг.
   Вероятно, уже даст осмысленный (пусть неполный) IR Space Marines.
2. **P0-b: межфайловая сборка** `.gst` + `.cat` (+ `catalogueLink`) — общий
   API `parse_system(files)` и объединённая символьная таблица.
3. **P1-a: `modifierGroup`** — закрыть тихую потерю модификаторов (корректность
   очков/лимитов) + громкие диагностики на всё ещё нераспознанное.
4. **P1-b: профили/характеристики** — новый домен-тип, отдельный трек (нужен для
   UI, не для валидации).
5. **P2** — display-текст (infoLinks/rules) — по мере надобности UI.

Между шагами имеет смысл повторять этот прогон: с громкими `*.unrepresented`
диагностиками (мета-находка) отчёт о пробелах станет автоматическим.
