## 1. Обзор и позиционирование

**Muster** — мобильное (+ веб) приложение-армибилдер для настольных варгеймов Games Workshop (Warhammer 40,000, Age of Sigmar, Kill Team и др.) на открытых каталогах данных **BSData** (формат BattleScribe). Аналоги — New Recruit и War Organ.

**Ценностное предложение (три опоры):**
1. **Лаконичный современный UI** — главный товар и дифференциатор. Бенчмарк чистоты — War Organ (его хвалят именно за это), но с шириной систем и функциями шаринга уровня New Recruit.
2. **Чистая валидация** — собственный движок вычисления правил с гранулярными *причинами* ошибок (чего про-игрокам не хватает в New Recruit) и корректным инкрементальным пересчётом (избегаем класса багов «порча валидации при add/remove детачмента»). Плюс **app-level слой правил-оверрайдов**, гасящий известные ложные ошибки *данных* (напр. understrength — их нельзя починить одним валидатором, т.к. информации нет в самих `.cat`; нужен наш надстроечный слой). См. §5.
3. **Offline-first во всём** — построение списков и (в будущих фазах) подсчёт очков за столом работают без сети; это же — killer-фича против лидера турнирного рынка (BCP требует постоянного интернета и падает на площадках с плохим wifi).

**Целевая ширина:** несколько игровых систем GW через универсальный движок BattleScribe (данные каждой системы приходят из своего `.gst`).

## 2. Цели и границы

### В объёме первого спека — Фаза 1 «Билдер»
- Регистрация/вход (опционально), аккаунт, авторизация по владельцу.
- Локальная БД + движок парсинга и валидации BattleScribe.
- Пайплайн доставки каталогов (подписанные пре-парсенные бандлы с CDN).
- Построение/редактирование армейских списков с живой офлайн-валидацией.
- Сохранение ростеров (именование, папки, дублирование).
- Импорт `.ros/.rosz` (миграция с BattleScribe/New Recruit).
- Шаринг **без регистрации**: копирование текстом, экспорт JSON (наш формат, файлом), QR; опционально share-ссылка через анонимную загрузку. *(Экспорт `.rosz` — опционально/позже, §13.)*
- Экспорт-вьюхи: PDF/карточки, текстовый формат, read-only viewer.
- **Reference Mode** (Фаза 1b): индекс по юнитам списка со сворачиваемыми датащитами — характеристики, профили оружия, тексты способностей, энхансменты, **детачмент-правило и стратагемы** — доступ в 1–2 тапа (в т.ч. во время игры). Данные уже есть в BSData-профилях; правовая рамка — §14.
- **Поля читаемости** (кастомное имя юнита / counts-as / заметки на юнит и ростер) — §6.
- Синхронизация списков между устройствами под аккаунтом (при наличии).
- Лаконичный UI ключевых экранов (мобайл; веб-каркас минимальный).

### Разбивка Фазы 1 (walking skeleton — снижение риска ядра)
- **Фаза 1a — walking skeleton:** одна система (напр. 40k), локально, без аккаунтов/синка; парсер + движок вычисления + минимальный билдер с валидацией. Цель — проверить самый рисковый элемент (**движок вычисления**) на реальных `.cat` как можно раньше.
- **Фаза 1b:** аккаунты + синк, импорт `.ros/.rosz`, шаринг/экспорт, несколько систем, лаконичный UI.

### Критерии приёмки Фазы 1 (черновик)
- Собрать легальный список 40k; поймать инлайн 100% нарушений из тест-набора (очки/min-max/слоты/детачмент) с корректными *причинами*.
- Ре-валидация правки ощущается мгновенной (перф-бюджет — §15).
- Импортировать реальный `.rosz` из BattleScribe/NR без потери данных.
- Поделиться списком и переоткрыть на другом устройстве (текст/JSON/файл/QR).

### Явно вне первого спека (отдельные спеки в след. фазах)
- **Фаза 2a:** «Проведение события» (жизненный цикл события, регистрация, чек-ин, таймер, паринги/бай) — см. §11.
- **Фаза 2b:** «Живой счёт» (цифровая замена бумажной турнирной карточки) — см. §11.
- **Фаза 3:** «Полные турниры» (авто-паиринги, ELO/ладдер, TO-инструменты) — см. §11.
- Полноценный визуальный дизайн (дизайн-система, hi-fi макеты) — отдельный проход после утверждения спека.
- Трекер коллекции/покраса — **сознательно вычеркнут** как непрофильный.

### Не-цели
- Не воспроизводим правовой статус закрытых источников (GW App, Wahapedia, чужие датасеты) — единственный источник данных армий — BSData.
- Не строим соцсеть (публичная лента, лайки/комменты) в обозримых фазах.
- **Нарративные/кампанийные форматы** (Crusade / Path to Glory / KT spec ops — персистентная прогрессия, Order of Battle, XP/honours) — **вне области**; продукт matched-play + турнирный. (Соответственно модель ростера — два состояния, без «кампанийного», §6.)

| Ось | New Recruit | War Organ | **Muster** |
|---|---|---|---|
| Ширина систем (BSData) | ✅ | ❌ только 40k | ✅ |
| Чистый современный UI | средне | ✅ | ✅ **главный козырь** |
| Качество валидации | наследует баги BattleScribe | ок (40k) | ✅ свой движок + слой оверрайдов |

## 4. Архитектура

**Стек:** единое TypeScript-монорепо. Обоснование выбора — сквозное переиспользование логики между клиентом, сервером, пайплайном и вебом; зрелая экосистема local-first-синка; один язык для малой команды. (Полный сравнительный разбор вариантов — Flutter, Dart-везде, KMP, Rust-native — вёлся при проектировании; TS победил по доминирующему критерию «один валидатор на клиенте и сервере».)

```
                    ┌─────────────────────────────┐
                    │   Catalogue/Scoring Pipeline │  крон: тянет BSData (GitHub),
                    │   (BSData → pre-parsed,       │  пре-парсит Rust-движком,
                    │    signed bundles → CDN)      │  подписывает, версионирует
                    └──────────────┬──────────────┘
                                   │ анонимно, read-only, verify signature
                                   ▼
   ┌──────────────┐        ┌──────────────────────────────┐        ┌──────────────┐
   │  apps/mobile │◄──────►│  packages/engine + domain    │◄──────►│   apps/api   │
   │  (RN/Expo)   │  local │  parser (Rust) + eval engine  │  auth  │   (Node)     │
   │  local-first │  eval  │  (TS) + shared types (Zod)    │  valid │  sync + auth │
   │  SQLite      │        │                               │        │  Postgres    │
   └──────┬───────┘        └──────────────────────────────┘        └──────┬───────┘
          │                                                                │
          └──────────────────── sync (local-first engine) ────────────────┘
```

### Пакеты/приложения монорепо
| Пакет/приложение | Технология | Роль |
|---|---|---|
| `packages/engine-parser` | **Rust** | Парсинг BSData XML → **разрешённый IR** (инлайн ссылок/импортов, ID-индексы, базовые стоимости). Безопасный разбор недоверенного ввода. Сервер/пайплайн (нативно, **без FFI**). Действительно лёгкий слой. |
| `packages/engine-eval` | **TS** | **Полноценный движок вычисления** правил: живая агрегация ограничений по скоупам + условные модификаторы + условия/repeat против состояния ростера. Плюс app-слой оверрайдов. Общий код: офлайн на клиенте + авторитетно на сервере. **Не «лёгкий» — это ядро легальности.** |
| `packages/domain` | TS + Zod | Единые типы, схемы, контракты API. |
| `apps/mobile` | React Native (Expo) | Клиент, local-first (SQLite) + синк. |
| `apps/api` | Node | Auth, синк, пайплайн, позже — события/счёт/турниры. |
| `apps/web` | Next.js | Веб-каркас, server-rendered шаринг-страницы, позже — TO-дашборд. |

### Принципы изоляции
- `engine-eval` — чистый TS без I/O; тестируется изолированно; исполняется и на клиенте, и на сервере.
- `engine-parser` (Rust) держится с **нулём небезопасных зависимостей**; граница «парсер → движок» — это **данные** (сериализованный IR в бандле), а не FFI.
- **Раннее покрытие тестами агрегации скоупов** (self/parent/ancestor/force/roster/category × shared × includeChild… × field) — риск корректности №1 по опыту реимплементаций.
- Клиент общается с локальной БД, не с сервером напрямую; синк — отдельная граница.
- Пайплайн контента отвязан от пользовательского бэкенда (разные контуры компрометации).

## 5. Движок: парсер (лёгкий) + движок вычисления (не лёгкий)

Разделение на две стадии верное и совпадает с тем, как устроены серьёзные реимплементации (напр. Phalanx от BSData). **Важная поправка по итогам техпроверки:** «валидатор» — это **не лёгкий чекер, а полноценный движок вычисления ограничений и модификаторов**. Лёгок только парсер.

| | Парсер (`engine-parser`, Rust) | Движок вычисления (`engine-eval`, TS) |
|---|---|---|
| Что делает | XML → **разрешённый IR** | оценивает легальность живого ростера |
| Тяжесть | **действительно лёгкий** | **тяжёлый — ядро легальности** |
| Недоверенный ввод | **да** | нет (свой список) |
| Место | Rust, сервер/пайплайн (без FFI) | TS, клиент (офлайн) + сервер (авторитетно) |

**Парсер (дёшево, запекаем в IR):** распаковка `.catz/.rosz` (это zip), XML→структуры, **разрешение графа ссылок** (entry links → shared entries, импорты каталогов faction→library→gamesystem, category/info links), ID-индексы и обратные карты, базовые стоимости, cycle-guard рекурсии. Это и есть реальная ценность парсера.

**Движок вычисления (дорого, НЕ лёгкое):**
- **Условные модификаторы** против живого состояния: `set/increment/decrement` меняют сами лимиты ограничений и стоимости в зависимости от других выборов (understrength `set min=0`, стоимость за модель и т.п.).
- **Агрегация ограничений с полной семантикой скоупов:** self/parent/ancestor/force/roster/primary-category × `shared` × `includeChildSelections/Forces` × field(selections|cost).
- **Условия/condition-group/repeat** против состояния — решают, какие модификаторы и ограничения вообще применяются.
- Это **state-dependent fixed-point**, а не таблица предвычисленных лимитов.

**Формат IR (граница Rust↔TS):** «скомпилированный AST + таблица символов», а НЕ «предвычисленные лимиты». Статику — запечь; узлы **constraint/modifier/condition/repeat нести как есть**, движок интерпретирует их против мутабельного состояния ростера на каждой правке.

**Дифференциатор «чистая валидация» — две честные части:**
1. **Гранулярный корректный инкрементальный движок** чинит класс *настоящих* багов движка (порча валидации при add/remove детачмента) и даёт причины ошибок детальнее, чем NR.
2. **App-level слой правил-оверрайдов** — для ложных ошибок *данных* (understrength), которые **нельзя** починить одним валидатором (информации нет в `.cat`). Наш слой гасит/переопределяет конкретные ограничения под app-флагом.

**Тот же слой оверрайдов = механизм house-rules (не только фикс багов данных).** Архитектурно «дисмиссить/переопределить конкретное ограничение» — это одно и то же действие, применяем ли мы его к системному ложному эрролу (understrength) или к пользовательскому «у нас в клубе так не играем». Поэтому оверрайд-слой обслуживает **оба** кейса: (а) встроенные системные оверрайды под app-флагом; (б) **казуальный/house-rules режим** — игрок вручную игнорирует конкретную ошибку валидации, список помечается «содержит хаус-рулы» (метка едет в шаринг/экспорт). Never-block уже заявлен (§9.2.5) — это шаг дальше по той же оси. Дёшево, т.к. механизм уже строится.

**Переиспользовать нечего:** движок New Recruit приватный, evaluator в Phalanx — незаконченная сложная часть, публичных open-source evaluator’ов нет (есть только читалки готовых `.ros`). **Пишем сами.**

**Гочи (в план реализации):**
- **Инкрементальный/скоуп-ограниченный пересчёт** — наивный полный ре-eval на каждый тап слишком медленный для реалтайм инлайн-ошибок на больших ростерах.
- **Порядок модификаторов и циклы** — внутренний алгоритм BattleScribe закрыт; итерируем до fixed-point и **реверс-инжинирим edge-cases эмпирически** против реального приложения.
- **shared vs per-instance** подсчёт и **корректность скоупов** — риск №1; тест-сьют строим рано.

*Открытая возможность (не Фаза 1):* при необходимости движок переносим в Rust/Wasm; интерфейс изолирован.

## 6. Модель данных и синхронизация

**Local-first + опциональная регистрация:**
- **Без аккаунта:** всё работает локально против SQLite; шаринг self-contained (текст/JSON/файл/QR).
- **С аккаунтом (по желанию):** облачный синк между устройствами; нужен для будущих событий/турниров.

**Синк:** local-first-движок (кандидаты — PowerSync / ElectricSQL / Replicache; выбор — на этапе плана). Работает офлайн, разрешает конфликты, синкает под аккаунтом. Тот же синк-механизм позже обслужит offline-first подсчёт очков за столом.

**Ключевые сущности Фазы 1:** `User (опц.)`, `Roster`, `RosterFolder`, `GameSystem`, `CatalogueVersion`. `Roster` хранит выбранную систему/каталог-версию, дерево выборов (юниты/опции), закэшированный результат валидации.

**Ростер — два состояния одной сущности (проектируем развилку сразу).** По итогам сценарного разбора статичная модель «Roster = снимок» недостаточна: турнир требует **иммутабельной заморозки** поданного листа отдельно от живого документа. Заложить развилку сейчас **дёшево**, ретрофитить статику потом — **дорого**. Два состояния:
1. **Living roster** — обычный редактируемый документ (дефолт Фазы 1). Мутабельный, синкается, мигрируется между каталог-версиями.
2. **Submission snapshot** (`ListSubmission`) — **иммутабельный** снимок `{roster_tree, catalogue_version, submitted_at, locked}`, создаётся при подаче на событие (Фаза 2). Правки после лока запрещены; отделён от living-ростера, чтобы правка синканутого объекта после дедлайна не подменяла поданный лист (дыра целостности). Раскрытие оппонентам — конфигурируемо.

**Поля читаемости на уровне выбора (Фаза 1, аддитивно):** кастомное отображаемое имя юнита (именованный персонаж / «Отряд Дэйва» / **counts-as** «считаем за X» с сохранением профиля правил), free-text заметка на юнит (лор/покрас) и на ростер. Нулевой риск — новые поля, едут в текст/PDF-экспорт; нужны и для печатной читаемости на ивенте, и для именованных персонажей.

**Версионирование и миграция ростеров:** каталоги обновляются (очки/правила меняются). `Roster` привязан к `CatalogueVersion`; при выходе новой версии **не ломаем молча** — помечаем ростер, показываем дифф («что изменилось / стало невалидно») и даём осознанную миграцию (аналог кросс-редакционного копирования у NR). Старую версию каталога держим доступной, пока ростер не мигрирован. **Исключение — турнир:** внутри активного события версию каталога фиксирует **событие** (§11.2a), автомиграция отключена до конца события — иначе поданные листы «поедут» на середине турнира.

## 7. Доставка контента: каталоги и определения скоринга

Единый **data-driven контент-пайплайн** обслуживает два вида данных: каталоги армий (Фаза 1) и определения скоринга (Фаза 2). Всё — **данные, а не хардкод**; обновление контента не требует релиза приложения.

**Механика:**
1. **Bundled snapshot** — приложение поставляется со снимком; работает офлайн с первого запуска.
2. **Проверка обновлений** — при наличии сети сверяет версии с манифестом, качает **дельту** с CDN.
3. **Верификация подписи** бандла (защита от подмены).
4. **Перепарс** (для каталогов — на стороне пайплайна Rust-парсером; клиент получает готовую модель).
5. **Ручной импорт** — пользователь может подсунуть файл каталога/ростера.

**Свойства:** анонимно, read-only, без PII; статический хостинг (near-zero attack surface). Компрометация каталогов и компрометация аккаунтов — **разные контуры**.

**Зависимость и риск:** BSData — community-проект; популярные системы (40k, AoS) обновляются быстро, нишевые отстают. Парсер оформлен отдельным модулем с чётким интерфейсом — при необходимости можно подключить второй источник (напр. Rosterizer) без переписывания ядра.

**Определения скоринга (Фаза 2) — гибрид:** официальные определения тянутся с подписанного CDN (GW GT, WTC, 10th/11th, AoS, KT), TO/клуб **кастомизирует поверх** (оверрайды кэпов/полей/конвертации/тайбрейков), скоуп — свой клуб/событие, неподписанные. Подробнее — §11.

## 8. Функционал Фазы 1 (детально)

- **Построение/редактирование списка** с **живой офлайн-валидацией**: очки, min/max, слоты категорий, правила детачмента; ошибки — инлайн, мгновенно.
- **Сохранение** ростеров: именование, папки/организация, дублирование; локально, синк при аккаунте.
- **Импорт `.ros/.rosz`** — миграция с BattleScribe/New Recruit/War Organ и всей экосистемы (их «ров»). *(Экспорт `.rosz` для интеропа с чужими билдерами/TTS — опционально/позже, §13.)*
- **Шаринг без регистрации:**
  - копирование как **текст**;
  - экспорт **JSON** (наш формат), в т.ч. файлом;
  - **QR-код** (данные внутри кода — self-contained);
  - опционально **share-ссылка** через анонимную загрузку (без логина), просмотр без аккаунта.
- **Экспорт-вьюхи:** PDF/«красивые карточки», текстовый формат, read-only viewer с правилами. Два профиля: **full (private, с рулзами)** и **opponent/TO-facing (public)**; PDF обязан быть **print-friendly** (ч/б, A4) — критерий приёмки, не только «красиво на экране».
- **Reference Mode** (Фаза 1b): индекс по юнитам списка → сворачиваемые датащиты (характеристики, профили оружия, тексты способностей, энхансменты) + отдельный экран **детачмент-правила и стратагем**; доступ 1–2 тапа, в т.ч. во время игры и (Фаза 2) из live-карточки счёта. Правовая рамка — §14.
- **Поля читаемости** на юните: кастомное отображаемое имя / **counts-as** / free-text заметки, плюс заметка на ростер; едут в текст/PDF-экспорт (§6).
- **Несколько систем GW** (40k, AoS, Kill Team…). *Оговорка:* «универсальный движок» надёжно покрывает очко-центричные системы; **процентные/equipment-point системы (Kill Team, Old World, Horus Heresy) требуют отдельной проверки движка** (§12.7, §13).
- **Аккаунты + синк** — опционально; офлайн-first.

## 9. UX-направление и ключевые экраны (функционально)

**North star — War Organ.** Основной ориентир UI/UX — **War Organ**: «super clean and accessible», заточка под скорость и ясность, единое цельное приложение, минимум хрома. Его эстетику и подход к взаимодействию берём за базу; у остальных заимствуем **точечно** — только там, где паттерн явно сильнее или где даже War Organ проседает (см. ниже). Мы не собираем «среднее по конкурентам» — мы делаем War Organ-уровня чистоту плюс адресные улучшения.

**Принципы:** минимализм, современная чистота, нативное ощущение, mobile-first (War Organ спроектирован iPad/mobile-first); плотность информации регулируемая (компактный вид по умолчанию). Ментальная модель — **«армия как документ», а не «дерево данных»**: War Organ/New Recruit читаются как армия и выигрывают в подходности; BattleScribe/Rosterizer выставляют пользователю модель данных и платят за это доступностью.

### 9.1 Разбор билдер-UX конкурентов (выжимка)

**База — War Organ.** Его паттерны берём как основу:
- **Предпросмотр стат-линии прямо в пикере добавления** — самая хвалимая интеракция во всей категории; решаешь с датащитом перед глазами. Ядро нашего add-flow.
- **Стоимость опций в очках — на месте выбора**: каждый апгрейд/вооружение показывает ±очки инлайн.
- **«Оставшиеся очки» как активный фильтр**: состояние сборки фильтрует каталог до того, что ещё влезает, — валидация как подсказка вперёд.
- **Разделение «армейское vs по юниту»** (страница детачмента): детачмент-правила/энхансменты — на экране детачмента, экран юнита — только про юнит.
- Заточка под **скорость и ясность**, компактность, минимум навигационного шума.

**Точечно у других — только где сильнее или где War Organ проседает:**
- **Реалтайм очки + реалтайм легальность** (New Recruit, GW App): тотал мгновенно; нелегальные выборы флагаются по ходу, без отдельного «проверить».
- **Пресеты/дефолтные лоадауты** (New Recruit): сохранить полный лоадаут и добавить в один тап.
- **Кастомные категории + свободный reorder** (New Recruit): организовать список как думает игрок.
- **Наши улучшения там, где War Organ проседает:** дублирование настроенного юнита в **один тап** (у WO слабое) и гранулярные *причины* валидации (см. анти-паттерны).

**Анти-паттерны (избегаем):**
- **Глубокие вложенные меню опций** — у New Recruit «четыре меню вглубь ради одного оружия». Конфиг юнита — в один прокручиваемый экран.
- **Add-then-inspect** (BattleScribe): без предпросмотра — churn «добавил/удалил».
- **Много тапов / захламлённая навигация** (GW App): каждый лишний тап в add-unit ощутим — это самое частое действие.
- **Выставление модели данных наружу** (Rosterizer): мощно, но отпугивает медианного игрока.
- **Медленная загрузка / тяжёлое состояние** (BattleScribe ~2 мин, Rosterizer тормозит): билдер обязан ощущаться мгновенным, особенно за столом.
- **Валидация либо слишком тонкая** (New Recruit для про), **либо погребена в тяжёлой оболочке** (GW App): цель — гранулярно *и* легко.
- **Слабое дублирование** (War Organ): дублировать настроенный юнит — один тап.
- **Неясный статус синка** (New Recruit mobile↔web): делаем синк-состояние читаемым.

### 9.2 Ключевые экраны и потоки (не пиксели)

1. **Список армий** — папки, поиск, дублирование, создание; на карточке ростера — очки + индикатор валидности.
2. **Экран билдера (IA):** липкий заголовок с **живым тоталом очков + кэп размера боя + единый индикатор валидности** (галочка / число проблем). Тело — армия как **сворачиваемые секции-категории** (по умолчанию battlefield role, но **переупорядочиваемые и рекатегоризируемые** пользователем). Отдельный **экран детачмента/армейского** (детачмент-правило + энхансменты) — вне потока юнита. Внизу секции — постоянное **«+ Добавить юнит»**. Без сырого дерева и без схемы наружу.
3. **Add-unit (оптимизируем безжалостно — самое частое действие):** «+ Добавить» → **поиск-first** пикер с фильтром по категориям, где **каждая строка = имя + очки + разворачиваемая стат-линия** (полный датащит без ухода из пикера) и **тумблер фильтра «оставшиеся очки»**. Добавление — один тап; добавление **сохранённого пресета** — один тап. Цель ≤2 тапов от списка до добавленного юнита. Без отдельного экрана «подтвердить».
4. **Unit-config:** юнит открывается **одним прокручиваемым листом** (модалка на мобайле, боковая панель на широких экранах), не цепочкой меню. На нём: **степпер числа моделей** сверху; опции — контролами под тип ограничения (**radio для «выбери 1», checkbox/степпер для «до N», disabled+причина для нелегального**); **у каждой опции ±очки инлайн**; энхансменты — только если легально для юнита; сабтотал юнита в шапке листа; **кастомное имя / counts-as / заметка** (лор·покрас) — сворачиваемое поле; **«Дублировать»** и **«Сохранить как пресет»** — один тап в оверфлоу.
4a. **Reference Mode (Фаза 1b):** из билдера/оверфлоу — режим «читалка армии»: индекс по юнитам → сворачиваемые датащиты (характеристики, профили оружия, тексты способностей, энхансменты) + отдельный экран **детачмент-правила и стратагем**. Цель — заменить пролистывание PDF/книг у стола; в Фазе 2 доступен из live-карточки счёта.
5. **Отображение валидации — двухуровнево, реалтайм, неблокирующе:** (а) **инлайн-бейджи** на проблемном юните/опции (красная точка + краткая причина по тапу) и (б) **сводка-чип в липком заголовке** («2 проблемы») → разворачивается в список со ссылками на нарушителей. Разделяем **hard-ошибки** (перебор очков, нелегальное количество — красный) и **soft-предупреждения** (нет варлорда — янтарный). Никогда не блокируем правку — подсказываем. Совмещаем реалтайм-ощущение New Recruit с гранулярностью *причин* BattleScribe.
6. **Плотность/режимы:** компактный вид по умолчанию (имя · модели · очки · бейдж); разворот юнита — сводка лоадаута; глобальный тумблер компакт/детально; **тёмная тема с первого дня**.
7. **Шаринг/экспорт** — единый лист выбора способа (текст/JSON/файл/QR/ссылка/PDF/viewer).
8. **Аккаунт** — вход/регистрация (опционально), **читаемый статус синка**, «удалить аккаунт».
9. *(Фаза 2)* **Live-карточка счёта** — по раундам, потоки объективов, кэпы, PIN-подтверждение.

### 9.3 Визуальная дизайн-система (тёмная тема v1)

Визуальная айдентика задана: тема **«Deep Space & Charcoal»**, стиль **Minimalist-Atmospheric** (много воздуха, атмосферные текстуры за резким чистым UI, «Tactical Clarity»). Канонические токены — **[тёмная](../../design/design-system-dark.md)** и **[светлая](../../design/design-system-light.md)** (фронтматтер = источник истины для кода). Ключевое:

- **Тема:** тёмная — v1 (совпадает с §9.2.6 «тёмная тема с первого дня»); **светлая — v1** (парная M3-схема, ролевой паритет 1:1, WCAG AA проверена; остаётся верификация на реальных экранах — open flags в файле). `typography/rounded/spacing` — общие для обеих тем.
- **Палитра (роли):** *Abyssal* (глубокий навы-чарколь) — канвас/глубокие слои; **Aurelian Gold — главный CTA и маркеры достижений**; **Crimson — live-индикаторы/алерты/вторичные CTA**; slate/серые — границы контейнеров без резких белых линий.
- **Типографика:** **Plus Jakarta Sans** (заголовки, тяжёлые веса), **Inter** (тело, дата-плотные списки/правила), **JetBrains Mono** (очки, номера столов, метаданные — «tactical readout»). Шрифты — локальные ассеты (офлайн-first, не CDN).
- **Раскладка:** 8px-база; 12-колоночная сетка на десктопе / одноколоночный стек на мобайле; контент центрируется в 1280px; на мобайле поля 20px, таблицы рефлоу в карточки.
- **Форма:** мягкое скругление 0.25rem (кнопки/инпуты 4px, карточки 8px, бейджи систем 2px — «военные жетоны»).
- **Глубина:** *тёмная* — тональные ярусы + 1px «травленые» (светлые) бордеры + glassmorphism (blur 12px, 70%) — не тяжёлые тени. *Светлая* — глубину несёт **мягкая cool-тень** (тональное расслоение слабое), «этчинг» инвертируется в тёмную хейрлайн, glass-непрозрачность 85–92% (детали — в файле светлой темы).
- **Компоненты:** золотые primary-кнопки; event-карточки (16:9 + mono-грид даты/очков/«мест осталось»); чипы систем (сине-glow 40k, золото/crimson ghost для AoS/KT); таймлайн расписания; пульсирующие «Live»/«Registration Open».
- **Алерты/статусы — severity кодируется ПАТТЕРНОМ, не только оттенком** (утверждено): **error = filled** («стоп»), **warn = золотой ghost**, **live = crimson ghost + пульс**, **info = перивинкл ghost**. Разводит error↔live с первого взгляда и убирает «золото на золоте»; все пары WCAG AA, 0 новых токенов; success намеренно без зелёного (золото/нейтральная галочка). Конкретные пары — в файлах дизайн-системы.

**Согласования при реализации (зафиксированы в файле дизайн-системы):** (1) «Primary CTA = Aurelian Gold» = токен **`secondary`**, не `primary` (перивинкл) — задать явный маппинг; (2) прозаические хексы elevation (`#0F172A` и т.п.) — иллюстрация, **источник истины — блок `colors`**; (3) семантика валидации §9.2.5 → hard-ошибка = `error`, soft-предупреждение = `secondary` (янтарь), live-пульс = `tertiary` (Crimson).

**Заметка по бренд-контексту:** описание дизайн-системы прямо апеллирует к Warhammer 40,000/Age of Sigmar. Это ориентир *эстетики/аудитории*, но публичная айдентика (имя, стор-листинг, маркетинг) обязана оставаться в рамках §14 — без ассетов/марок GW. Особое внимание — предложенному имени (см. §13/§14).

**Hi-fi макеты и интеракции** — отдельным проходом (интерактивные макеты в браузере) поверх этих токенов; они превратят §9.1–9.3 в конкретные экраны.

## 10. Безопасность (threat model)

**Модель нарушителя:** (1) посторонний, присылающий вредоносный файл (`.rosz/.cat`) или открывающий share-ссылку; (2) аутентифицированный пользователь, тянущий/меняющий чужие данные; (3) участник турнира, накручивающий счёт (Фаза 2); (4) сеть/MITM; (5) компрометация зависимостей. **Принципы:** least privilege, defense in depth, сервер — источник истины, минимизация данных, secure-by-default.

### 10.1 Разбор недоверенных файлов (`.rosz/.catz/.cat/.gst`) — приоритет №1
«Memory-safe (Rust)» защищает от порчи памяти, но **не** от логических атак. Обязательно:
- **XML:** отключить DTD и внешние сущности (**no XXE**); запретить entity expansion (**no billion-laughs**); лимиты глубины/размера.
- **Zip (`.rosz/.catz` = zip):** лимит распакованного размера и ratio (**anti zip-bomb**); защита от `../` в путях (**anti zip-slip**); ровно один ожидаемый XML.
- **Ресурс-лимиты:** max размер файла, max время парсинга, max узлов; таймаут/квоты в пайплайне.
- Парсинг — в изолированном воркере/процессе с ограниченными правами; сбой не рушит сервис.

### 10.2 Аутентификация и сессии
- **Managed-провайдер** (Clerk/Supabase Auth/Auth0), не самопис; настраиваем харднинг.
- **Против enumeration:** единообразные ответы на логин/регистрацию/сброс.
- Верификация email; политика паролей / проверка на утёкшие (HIBP); опц. **MFA**.
- Сессии: короткоживущий access + ротация refresh; **отзыв** сессий; экспирация; хранение только в **Keychain/Keystore** (не AsyncStorage).
- **Rate-limit и лок-аут** на логин/сброс/подтверждения сверх общего throttling.

### 10.3 Авторизация (object-level)
- **Проверка владельца на КАЖДЫЙ ресурс** (ростер/папка/событие/матч) — против **IDOR/BOLA**; централизованный authz-слой, не ad-hoc.
- **Неугадываемые ID** (UUID/ULID), не последовательные.
- (Фаза 2) роли **player/TO/club-owner**; изоляция клубов (мультитенант); защита от эскалации роли; аудит админ-действий.

### 10.4 Синхронизация и хранение на клиенте
- Authz на синк: пользователь синкает только своё.
- Целостность офлайн-очереди; anti-replay/подмена; сервер — авторитет.
- Чувствительное at-rest — шифровать; **deep-links валидировать** (anti open-redirect / param-инъекция); учитывать jailbreak/root (не хранить лишнего).

### 10.5 Анонимный шаринг и веб-страницы
- Share-link (анонимная загрузка): **неугадываемые ID**, лимит размера, **экспирация**, rate-limit, анти-абуз/модерация, никакого исполняемого контента.
- Веб-страницы (server-render): **строгое экранирование** user-controlled (имена/заметки) → **anti-XSS**; **CSP** + HSTS; без SSRF.

### 10.6 Контент-пайплайн и секреты
- Бандлы (каталоги/официальный скоринг) — **подписаны**; клиент верифицирует; невалидная подпись → откат на прошлую версию.
- Авторские определения TO — **неподписаны**, скоуп клуб/событие, вне глобального доверия.
- **Ключ подписи** — в KMS/секрет-хранилище, ротация; CI-секреты изолированы; никаких секретов в репо/логах.

### 10.7 Транспорт, DoS, логирование
- TLS везде; опц. cert-pinning на мобайле; HSTS.
- Rate-limit **не только на auth:** парсинг, анонимная загрузка, синк, общий API.
- Security-логирование **без PII**; скраб крашрепортов; алерты на аномалии.

### 10.8 Supply-chain
- **npm:** lockfile, Dependabot/SCA, `npm audit` в CI, provenance, минимум зависимостей (в парсере — по возможности 0).
- **Rust:** `cargo audit`/`cargo deny`, pinning; минимум `unsafe`.
- Подпись релизов; SLSA-provenance где возможно; BSData-контент — недоверенный (см. 10.1).

### 10.9 Приватность / комплаенс
- **Минимизация PII** (email + никнейм); карта данных; retention-политика.
- **Права субъекта:** удаление аккаунта, экспорт данных (портируемость), согласия (GDPR/152-ФЗ); DPA с провайдерами; процесс на утечку.
- Аналитика — opt-in, без PII.

### 10.10 Фаза 2 — целостность счёта
- Двойное подтверждение обоих игроков; **PIN стойкий + anti-брутфорс**; нельзя подтвердить за оппонента; anti-replay.
- Сервер **реконсилирует** самозаявленные счета; **неизменяемый аудит-лог** оверрайдов/сбросов админа.

## 11. Roadmap: Фазы 2–3

Фазы 2–3 не входят в первый спек (получат свои детальные спеки), но зафиксированы здесь, чтобы **фундамент Фазы 1 их учитывал** (бэкенд, роли, синк, обобщённая модель контента).

### Фаза 2 — «События»: 2a Проведение + 2b Живой счёт

**Мотивация:** сейчас счёт ведут карандашом на бумаге и сдают администратору. Заменяем это на in-app ввод результата с передачей админу; админ ведёт статистику в приложении. Контекст — **и турниры, и постоянная клубная лига**.

**Модель:** `Клуб → Сезон/Лига → Событие → Раунд → Матч`, с **долгосрочными профилями игроков** и клубным ладдером. Роли: **игрок / администратор (TO) / владелец клуба**.

**Почему две под-фазы (итог разбора со стороны TO):** *провести событие* ≠ *записать счёт*. Вся операционка **вокруг** счёта (регистрация, capacity, чек-ин, тайминг, паринги, бай) не менее важна и её нельзя откладывать в Фазу 3 — иначе Фаза 2 «записывает счёт», но не «проводит событие». Поэтому **2a (проведение)** идёт перед **2b (счёт)**.

#### Фаза 2a — Проведение события (операционный слой TO)

**Жизненный цикл события (state-machine):** `draft → registration_open → registration_closed → lists_locked → in_progress(round N) → finished`.

**Сущности:**
- **`Event`** — `capacity, registration_opens/closes_at, list_deadline_at, entry_fee?, venue_tables[], pinned_catalogue_version, scoring_def_version, tiebreak_stack`.
- **`Season`/`League`** над `Event` — окно дат, набор событий, конфиг зачёта (`count_best_n, min_events, attendance_points`), перенос/сброс рейтинга между сезонами. Без этого уровня «клубная лига» (заявленная мотивация) не моделируется.
- **`Registration`** (entry, отдельно от подачи ростера) — `event_id, player_id, status(registered|waitlisted|checked_in|active|dropped|no_show), paid, roster_id?, submitted_at, checked_in_at`. Вейтлист = упорядоченная очередь с авто-продвижением при дропе до старта.
- **`Round`** — `number, status(pending|active|finished), started_at, time_limit, ends_at`.
- **`Match`** — пара + `is_bye` (фиктивный оппонент, конфиг VP/результат бая).

**Ключевое:**
- **Флаг оплаты — учётная галочка, а НЕ платёжная система.** `Event.entry_fee?` (справочно, сколько взнос) и `Registration.paid(bool)` — это лишь **статус** «заплатил/нет», который TO проставляет вручную и по которому фильтрует допуск (не пускать неоплативших в пары/чек-ин). Muster **не** проводит платежи, не берёт комиссию, не интегрируется с эквайрингом — деньги (взнос за стол/призы/аренду) идут **мимо приложения**, между игроком и TO. Это осознанно: встроенный процессинг потянул бы PCI-комплаенс, эскроу, возвраты — тяжесть, несовместимую со свободным приложением. **Разграничение:** §17 «без пейволлов» — про функции *самого Muster* (все бесплатны); entry fee — деньги за участие *в турнире*, вне области ПО. Это разные вещи.
- **Событие пиннит версию каталога армий** — все листы валидируются против неё, автомиграция §6 отключена до конца события (иначе поданные листы «поедут» на середине турнира).
- **Раунд-таймер** — серверный `ends_at` + локальный офлайн-отсчёт; TO-действия `start / extend(+N мин) / end_now`; поток «время вышло → фиксируем текущий прогрессивный VP как финал». Тайминг — ядро работы TO, поэтому в 2a, а не в Фазе 3.
- **Бай + семантика SoS** — бай исключается из знаменателя SoS; правило SoS для `dropped/no_show` оппонентов **задокументировано** (учитывать только сыгранные / замораживать запись на момент дропа). Незаданная семантика SoS на бае/дропе = гарантированные споры у стойки.
- **Чек-ин + QR/код-вступление в событие + walk-in** — снижение трения у стойки; гостевой профиль, конвертируемый в аккаунт позже.
- **TO-proxy ввод** — TO вводит результат/лист за пару/игрока (сел телефон / бумажный лист / игрок без устройства); помечается в аудите; обходит dual-confirm под правом TO. **Штатный режим**, не только аварийный — иначе dual-confirm блокирует реальность зала.
- **Broadcast-канал** — объявления TO → лента события; экран игрока **«Мой турнир»** (текущий раунд, оппонент, стол, таймер, история).
- **Точечный unlock матча** задним числом (отдельно от «сброса раунда») + каскадный пересчёт таблицы + аудит.
- **Со-TO** на одном событии + политика разрешения конфликтов параллельных офлайн админ-правок (синк-конфликты §6 — но для админ-действий).
- Паринги в 2a — **ручные** (авто — Фаза 3). Честная граница: ручной паринг швейцарки на 24+ трудоёмок и ошибкоопасен → 2a пригодна для малых/round-robin форматов, крупная швейцарка ждёт авто-паринги Фазы 3.

#### Фаза 2b — Живой счёт (цифровая замена турнирной карточки GW)

**Поток «игрок → подтверждение → админ»:**
```
Игрок ведёт live-tally (offline, локально)
        │  конец раунда/игры
  оба игрока подтверждают PIN ──► совпало ──► lock ──► синк ──► таблица пересчитывается
        │                          
     расхождение ──► флаг администратору (оба значения рядом) ──► TO оверрайд/сброс раунда
```

**Заимствованные паттерны (из ресёрча BCP/Longshanks/NR/Tabletop Battles):**
- Полный per-game захват счёта (W/L/margin — **производные**, не вводятся).
- Двойное PIN-подтверждение обоих игроков (анти-чит BCP), но с **явным** экраном расхождения (у BCP он мутный — наше улучшение).
- TO override + сброс раунда + точечный unlock — обязательные «аварийные выходы».
- **Player-side коррекция/undo/история** своего tally **до** финального lock (после — эскалация TO). Реальность: заметил на R4 недописанные 5 VP за R2.
- **Live-CP трекер** — CP как **живой ресурс** за столом (старт, +1/период, лог трат стратагем), а не только итоговое поле.
- Настраиваемый **упорядоченный стек тайбрейков** (дефолт: запись → Strength of Schedule → сумма VP/margin → head-to-head), отдельно для паирингов и финальной таблицы; **финальный tie-break** на случай полной ничьи после всего стека.
- Живой счёт **по ходу партии**, не только в конце раунда.
- Интеграция: вступление в событие **валидированным списком** — снимает *механическую* проверку, но **не заменяет ручное ревью TO** (прокси, Legends, соответствие пакету): engine-valid ≠ отсутствие ревью (§13).
- **Колода секондари/так-опов как контент** (пикер fixed vs tactical из поставляемых карт, §7), а не свободный ввод `name`.
- **Не пейволлить** прозрачность (списки/таблицы видны).

**Killer-фича:** **offline-first ввод и подтверждение счёта** с фоновым синком — прямое закрытие боли лидера (BCP требует постоянного интернета и падает на площадках).

#### Модель счёта (data-driven, обобщённая)

Карточка счёта = **декларативное определение скоринга** (JSON, Zod-валидируемое), интерпретируемое движком счёта. Меняется публикацией новой версии определения — **без релиза**.

**Три грани данных** (покрывают карточки WTC 2025 и GW GT, обе редакции):
1. **Матч-игрок (итоги):** `faction, opponent_id, round, table, cp, painting(0/10), primary_total, secondary_total, total_vp(0–100), result(W/L/D), margin, game_points(шкала события), sportsmanship?`
2. **По периодам (battle round 1–5):** `primary_vp, secondary_vp` (прогрессивный счёт).
3. **По секондари-карте:** `name, mode(fixed|tactical), vp, round, status(active|kept|scored|discarded)`.

**Обобщения (закладываются сразу):**
- **Edition-config таблица** для кэпов: 10th = 50/40/15/10, 5 раундов; 11th ≈ 45/45/15/10. Кэпы — данные, не код.
- **«Период» вместо «battle round» + «потоки объективов» вместо «primary/secondary»** → одна схема обслужит 40k, AoS (5 раундов, 6+4, дабл-тёрн) и Kill Team (Turning Points, Crit/Tac Ops).
- **Секондари — hand переменной длины** (11th: «тяни 2, оставляй несыгранные»), не фикс-2-слота.
- **Сырой VP/margin отделён от очков события** (W/L/D против 20-очковой шкалы WTC).
- **`opponent_id` — first-class** (швейцарка + Strength of Schedule).
- **`painting` — из scoring-config, не хардкод `0/10`** (пакеты различаются: WTC покрас в game points не входит; часть GT-пакетов считает иначе/не считает).
- **Battle size — вход валидации, а не только кэп очков** (§9.2/§13): в 10th размер боя управляет лимитом энхансментов и числом детачментов; событие задаёт tier, движок применяет обусловленные ограничения.

**Минимум для таблицы (на игру):** `{player, opponent, round, result, total_vp, margin}`.

**Редактирование карточек — гибрид:** тяга официальных определений с CDN + оверрайды TO поверх; версия определения **фиксируется при создании события** (правка на середине турнира не портит введённые счета).

#### Админ-панель (TO / владелец клуба)

Ранее раздел был проговорён лишь через роли — здесь детализируем. Админ-панель — **отдельный режим приложения** (не отдельное приложение): те же RN-клиент и Next.js-веб, но экраны и права видит только `TO`/`club-owner`. Веб — основная площадка админа (широкий экран, много таблиц), мобайл — «на площадке в руках».

**Роли и права:**
- **`club-owner`** — владелец пространства (клуб/лига): управляет участниками и ролями, создаёт события, ведёт постоянный ладдер, видит аудит.
- **`TO` (администратор события)** — назначается на конкретное событие: настройка, регистрация/приём списков, тайминг раундов, паринги (ручные — 2a, авто — Фаза 3), ввод/разрешение счёта, публикация таблицы. Права **скоупятся событием** — TO одного турнира не трогает другой. Возможен **со-TO**.
- **`player`** — участвует, вводит свой счёт; админских экранов не видит.
- Изоляция клубов (**мультитенант**), защита от эскалации роли, **неизменяемый аудит-лог** админ-действий (§10.3, §10.10).

**Ключевые экраны/потоки админа:**
1. **Дашборд события** — статус в реальном времени: текущий раунд + **таймер**, счётчик чек-инов (X/Y пришло), «не подал список / не оплатил / на вейтлисте», сколько матчей `введено / подтверждено / в расхождении / не начато`; кнопки `start/extend/end` раунда, «опубликовать таблицу».
2. **Настройка события** — система/редакция, **battle size (кэп очков + обусловленные правила)**, число раундов, **capacity**, окна регистрации, **дедлайн подачи списков**, **event-pinned версия каталога**, **тумблеры Legends / allies-policy (+ally-cap)**, entry-fee-флаг, **выбор определения скоринга** (официальное с CDN + оверрайды кэпов/полей/конвертации), **упорядоченный стек тайбрейков**, видимость (списки/таблица публичны или нет). Версии определения и каталога фиксируются при создании.
3. **Регистрация и приём списков** — очередь записавшихся (вейтлист/чек-ин/оплата) + поданные ростеры со статусом валидации. Движок снимает **механическую** проверку, но **TO делает ручное ревью** (прокси, Legends, соответствие пакету) — approve/reject с причиной; **«принят вне системы»** для бумажных/чужих билдеров; блок на невалидных.
4. **Мониторинг раунда** — таблица всех матчей раунда, каждый со статусом подтверждения; проваленные dual-confirm подсвечены сверху.
5. **Разрешение расхождений** — очередь флагов: оба самозаявленных значения рядом; действия TO **override / сброс раунда / запрос переввода**; каждое пишется в аудит.
6. **Таблица и тайбрейки** — живой пересчёт (запись → SoS → сумма VP/margin → head-to-head, настраиваемо), ручной пересчёт, публикация; экспорт (CSV/публичная веб-страница §4).
7. **Игроки / Сезон-Лига (`club-owner`)** — профили, история встреч (для SoS/паирингов), **уровень Сезон/Лига** над событиями (окно дат, зачёт `best_n`/`min_events`/attendance, перенос рейтинга), постоянный клубный ладдер; назначение TO (в т.ч. **со-TO**).
8. **Аудит-лог** — неизменяемая лента всех оверрайдов/сбросов/unlock/правок ролей (кто/что/когда).
9. **TO-proxy ввод** — результат/лист за пару/игрока без устройства; помечается proxy в аудите (§2a).
10. **Broadcast** — объявления TO → лента события; в паре с игровым экраном «Мой турнир».

**Оффлайн для админа:** ввод и разрешение счёта работают **offline-first** (killer-фича) — TO на площадке с плохим wifi продолжает вести событие, синк догоняет фоном; параллельные админ-правки со-TO разрешаются политикой конфликтов (§2a); публикация таблицы требует консистентного состояния (сервер реконсилирует, §10.10).

**Границы фаз:** **2a** — жизненный цикл события, регистрация/чек-ин, таймер, **ручной** паринг + бай, TO-proxy, broadcast. **2b** — счёт/подтверждение/таблица. **Авто-паринги** (швейцарка/элиминация), пуши, «следующий раунд в один тап» и ELO/ладдер-аналитика — Фаза 3.

### Фаза 3 — «Полные турниры»
- Авто-паиринги: швейцарка, группы, single/double elimination; ручной оверрайд.
- ELO/клубный ладдер, аналитика (винрейт по фракциям/юнитам).
- Пуши + «следующий раунд в один тап» (самый хвалимый UX BCP); шахматные **game clock** per-player (WTC-требование), не только раунд-таймер 2a.
- Расширенный TO-UI (BCP критикуют за «уродливый» интерфейс — возможность обойти).

## 12. Стратегия тестирования

Пирамида: много юнитов (особенно движок), меньше интеграционных, тонкий слой E2E на критичных потоках, плюс отдельные классы **security / property / perf**. CI-гейты блокируют мёрдж при падении.

### 12.1 Юнит / компонент
- **`engine-eval` (TS) — ядро, макс. покрытие:** тест-сьют на **агрегацию скоупов** (риск №1) — рано; модификаторы (`set/increment/decrement`, условные, циклы); условия/condition-group/repeat; app-слой оверрайдов (understrength); детерминизм fixed-point.
- **`engine-parser` (Rust):** реальные `.cat/.gst`; разрешение ссылок/импортов; cycle-guard.
- **`domain` (Zod):** валидация схем, граничные значения.

### 12.2 Интеграционные
- Парсер→IR→движок: сквозной путь на реальном каталоге.
- Импорт `.rosz` → внутренняя модель (совместимость с BattleScribe/NR/WO).
- Синк-движок против бэкенда (офлайн-очередь, конфликты).
- **Контрактные тесты** клиент↔сервер через общие Zod-схемы.
- Пайплайн: BSData→бандл→подпись→клиент верифицирует.

### 12.3 E2E (отсутствовали — добавляем)
- Мобайл: **Maestro/Detox**; веб: **Playwright**.
- Сценарии: собрать список→валидация→сохранить→поделиться→реимпорт; регистрация/логин/сброс; синк между двумя клиентами; **офлайн→онлайн**; миграция ростера на новую версию каталога.

### 12.4 Security-тесты (отдельный класс)
- **Парсер:** XXE, billion-laughs, zip-bomb, zip-slip, обрезанный/битый файл, лимиты размера/глубины/времени — **корпус вредоносных фикстур**.
- **Authz/IDOR:** юзер A не читает/не меняет ресурс юзера B; неугадываемость ID.
- **Auth:** enumeration, брутфорс/лок-аут, повторный сброс-токен, истёкший/отозванный токен.
- **Веб:** XSS через имена/заметки; проверка **CSP**.
- **Анонимный шаринг:** неугадываемость/экспирация share-ID, лимиты, анти-абуз.
- *(Фаза 2)* брутфорс PIN, подтверждение за оппонента, целостность аудит-лога.
- CI: **SAST** + зависимостный скан; периодический **DAST/фаззинг**.

### 12.5 Property-based (движок)
- **fast-check:** случайные ростеры → инварианты (сумма стоимостей, границы кэпов, идемпотентность ре-eval, отсутствие крашей).

### 12.6 Перф-тесты
- Бюджеты: ре-валидация правки на ростере ~2000 очков **< ~50 мс** (инкрементально); cold start **< ~2 с**; загрузка/переключение каталога. Регресс-бенчмарки в CI.

### 12.7 Каталог edge-cases (обязательные фикстуры)
- **Движок:** shared-entry по всему ростеру; рекурсивные/связанные entries; `set min=0`; циклы модификаторов; пустой ростер; перебор на +1; слот ровно на лимите; understrength; мульти-детачмент; **союзный детачмент + ally-cap**; **battle-size tier меняет лимиты** (энхансменты/детачменты на Incursion/Strike Force/Onslaught); **процентные/equipment-point системы** (KT equipment points, Old World Core/Special/Rare %, HH FOC-%); **house-rules оверрайд** (игрок дисмиссит ошибку); огромный ростер (перф); каталог, который не парсится.
- **Импорт:** битый/обрезанный zip, не-UTF8, entity-laden XML, ростер на неизвестную версию каталога, ростер из другой системы.
- **Синк:** одновременная правка одного ростера офлайн на 2 устройствах; delete vs edit; частичный сбой; долгий офлайн; расхождение часов.
- **Каталог-обновление:** невалидная подпись; частичная закачка; даунгрейд версии; битый бандл; переполнение хранилища.
- **Шаринг:** **QR-ёмкость** (большой ростер не влезает → компрессия/фолбэк на ссылку); экспирация ссылки; юникод/эмодзи в именах.
- **Auth:** истёкший токен посреди действия; отозванный токен.
- *(Фаза 2)* dual-confirm расхождение; реконсиляция счёта; корректность тайбрейков на edition-config.

### 12.8 Прочее и CI-гейты
- Матрица устройств/ОС/браузеров; **a11y**-тесты ключевых потоков.
- **Тест-данные:** реальные BSData-каталоги + golden-ростеры + корпус вредоносных файлов как фикстуры.
- **CI-гейты:** порог покрытия движка; зелёные security-сканы; фаззинг; перф-регрессии — блокируют мёрдж.

## 13. Открытые вопросы и риски

1. **11-я редакция 40k только вышла (июнь 2026)** — мисс-паки/кэпы ещё стабилизируются. Мап валидации/скоринга держим edition-параметризованным; кэпы сверить по Chapter Approved 2026 и первому 11th-ed WTC/GT пакету. Зрелая база для тестов — 10th ed.
2. **Выбор local-first синк-движка** (PowerSync/ElectricSQL/Replicache) — на этапе плана; критерии: офлайн-конфликты, RN-поддержка, самостоятельный хостинг.
3. **Auth-провайдер** (Supabase Auth / Clerk / Auth0) — managed вместо самописа; выбрать на этапе плана.
4. **Размещение движка в Rust/Wasm на клиенте** — осознанно отложено; интерфейс изолирован для возможного переноса.
5. **Имя.** *Решено:* «Astronomican Lighthouse» — **внутренний коднейм** (разработка + дизайн-система), в публичной айдентике **не используем** («Astronomican» — термин вселенной GW, конфликт с §14). **Публичное имя — TBD к релизу**, GW-безопасное; кандидаты Muster / Levy; до релиза — проверка домена и товарного знака (классы ПО/игры).
6. **Зависимость от BSData** — риск структуры/актуальности чужого репозитория; митигируется модульным парсером и возможностью второго источника.
7. **Движок вычисления — крупнейший технический риск.** Корректная валидация BattleScribe = полноценный движок (модификаторы/скоупы/условия); переиспользовать нечего (NR приватный, Phalanx незакончен). Митигация: walking skeleton (Фаза 1a) проверяет ядро рано; ранний тест-сьют на скоупы; эмпирический реверс порядка модификаторов против реального приложения.
8. **Экспорт `.rosz`** — **отложен как опциональный** (интероп с чужими билдерами/TTS): переоткрытие требует точного повторения схемы BattleScribe (GUID/entry links). Импорт `.rosz` — в Фазе 1; наши форматы (текст/JSON/QR/ссылка/PDF) закрывают шаринг MVP.
9. **Локализация** — *решено:* i18n с первого дня, старт EN+RU (§15).
10. **Монетизация** — *решено:* полностью бесплатно, без рекламы (§17).
11. **Семантика SoS на бае/дропе/no-show** — незаданная = споры у стойки. Определить формулу до Фазы 2a (§11.2a): бай вне знаменателя SoS; поведение для dropped/no-show оппонентов задокументировать.
12. **Engine-valid ≠ TO-review** — движок снимает механику, но ручное ревью списков (прокси/Legends/пакет) остаётся; UX приёма списков (§11 экран 3) обязан это поддерживать, а не считать «TO не проверяет».
13. **Интероп с Best Coast Pairings** — соревновательный мир живёт в BCP; позиция «offline-first убьёт BCP» рискованна для адопшена. Зафиксировать интероп-путь (импорт/экспорт листа) и рамку «дополняем, а не заменяем» — на этапе плана Фазы 2.
14. **Процентные/equipment-point системы** (Kill Team, Old World, Horus Heresy) на «универсальном» движке — оптимизм требует проверки: отдельный тест-набор наравне с 40k (§12.7); «универсальность» подтвердить эмпирически, не декларативно.
15. **Battle size как драйвер правил** (не только кэп очков) — лимиты энхансментов/детачментов зависят от tier; вход валидации, фикстуры на Incursion/Strike Force/Onslaught (§12.7).
16. **Сравнение двух листов (diff) и fork шаренного листа в библиотеку** — базовая рутина листбилдинга/нет-листинга, в текущем UX не описана; кандидаты в Фазу 1b (решить на этапе плана).

## 14. Легал / IP (риск, требует проработки)

- **Лицензия BSData:** данные ведутся сообществом под открытой лицензией; подтвердить условия редистрибуции через наш подписанный CDN (пре-парсенные бандлы = производная) и указать атрибуцию.
- **IP Games Workshop:** GW агрессивно защищает торговые марки и контент; армибилдеры (BattleScribe/NR) ходят по грани, избегая ассетов GW и используя только community-данные. Muster: **не** используем логотипы/арт/официальные книги-тексты GW; только BSData; дисклеймер о непринадлежности к GW; осторожность с названиями в сторах. Перед релизом — юридическая консультация.
- **Имя/коднейм:** «Astronomican Lighthouse» — только **внутренний коднейм** (репо, дизайн-система, dev-сборки). «Astronomican» — термин сеттинга GW → **в публичном имени, стор-листинге, домене и маркетинге не появляется**. Публичная айдентика — GW-безопасное имя (§13.5); проверка ТЗ/домена до релиза.
- **Reference Mode и рамка «текстов правил» (§8/§2):** различаем (а) **ассеты/книги GW** — не трогаем; (б) **содержимое BSData-профилей** (характеристики, профили оружия, краткие тексты способностей/энхансментов, стратагемы) — показываем, т.к. это ровно те данные, что уже несёт `.cat` и на которых работает движок; отображение не расширяет объём данных, лишь читает уже загруженное. Reference Mode **не** дублирует полные тексты из книг сверх того, что есть в каталоге. Границу подтвердить на юридической консультации вместе с редистрибуцией BSData.

## 15. Нефункциональные требования

- **Производительность:** ре-валидация правки на ростере ~2000 очков — бюджет < ~50 мс (инкрементально); cold start < ~2 с; переключение каталога — без «спиннера смерти» (антипаттерн BattleScribe ~2 мин).
- **Хранилище:** офлайн-бандлы нескольких систем GW объёмны — ленивая загрузка систем по требованию, а не всё сразу.
- **Локализация/i18n:** **i18n-каркас с первого дня; запуск на EN + RU.** Строки вынесены, готовность к добавлению языков.
- **Доступность (a11y):** контраст, размеры тап-таргетов, screen reader на ключевых потоках.
- **Телеметрия/крашрепортинг:** приватность-совместимо (без PII, opt-in аналитика); крашрепорты для стабильности.

## 16. Обработка ошибок

- **Импорт:** битый/несовместимый `.rosz` → понятная ошибка, частичное восстановление где можно, без молчаливых потерь.
- **Верификация подписи бандла:** провал → не применять, остаёмся на прошлой валидной версии, уведомляем.
- **Парсер на кривом каталоге:** изоляция сбоя, лог, не рушим приложение (фаззинг в тестах).
- **Синк:** офлайн-очередь, ретраи, читаемый статус, разрешение конфликтов; никогда не терять пользовательские правки.
- *(Фаза 2)* **Dual-confirm расхождение:** first-class видимое состояние → эскалация админу.

## 17. Монетизация

**Решение: полностью бесплатно, без рекламы** (модель War Organ) — максимум лояльности аудитории. Следствия для дизайна: **никаких пейволлов** на фичи (включая синк/облако/турниры) — все функции доступны всем. Стоимость сервера/синка держим низкой (managed-сервисы, статический CDN); при необходимости позже — добровольный supporter **без функциональных привилегий** (чтобы не вводить пейвол).

## 18. Локальная разработка и запуск (dev experience)

Как поднять и тестировать проект на своей машине. Цель — **`clone → одна команда → работает`**, и чтобы **Фаза 1a не требовала бэкенда** (движок + мобайл запускаются без Postgres/CDN).

**Пререквизиты:** Node LTS + **pnpm** (workspaces), **Rust toolchain** (парсер), **Docker** (локальный Postgres — только для api/синка), **Expo CLI** (мобайл). Монорепо-оркестрация — **Turborepo** (кэш сборок/тестов, граф зависимостей пакетов).

**Раскладка запуска по пакетам:**
| Что | Команда (ориентир) | Нужен бэкенд? |
|---|---|---|
| Движок (тесты, TDD ядра) | `pnpm test --filter engine-eval` | нет |
| Парсер (Rust) | `cargo test` в `engine-parser` | нет |
| **Мобайл (Фаза 1a)** | `pnpm --filter mobile expo start` → Expo Go / симулятор iOS / эмулятор Android | **нет** (локальный SQLite + бандл-снапшот) |
| API + синк (Фаза 1b) | `docker compose up` (Postgres) → `pnpm --filter api dev` | да |
| Веб-каркас | `pnpm --filter web dev` (Next.js) | опц. |
| Пайплайн контента (локально) | скрипт: тянет BSData → парсит → **самоподписанный** dev-бандл в локальный CDN-стаб | нет |

**Данные для разработки:** в репо — **зафиксированный snapshot BSData** (не тянем сеть на каждом запуске) + **golden-ростеры** + **корпус вредоносных фикстур** (§12.7). Бандлы в dev подписаны **dev-ключом** (не прод-KMS, §10.6); клиент в dev-режиме доверяет dev-ключу.

**Конфигурация:** `.env.example` в каждом приложении; секретов в репо нет (§10.6). Managed-провайдеры (auth, синк) — через локальные ключи разработчика или их эмуляторы/локальные режимы, чтобы Фаза 1b поднималась без облака где возможно.

**Быстрый старт (эскиз):**
```
pnpm install                      # весь монорепо
pnpm test --filter engine-eval    # ядро — TDD без окружения
pnpm --filter mobile expo start    # Фаза 1a: билдер на устройстве/симуляторе
# Фаза 1b (по мере надобности):
docker compose up -d              # Postgres
pnpm --filter api dev             # auth + синк
```

**CI ↔ локально — паритет:** те же команды (`test`/`lint`/`typecheck`/`build`/security-сканы, §12.8) гоняются и локально, и в CI через Turbo — «зелено у меня» = «зелено в CI». E2E (Maestro/Detox/Playwright, §12.3) — отдельной командой, т.к. тяжелее.

## 19. Резюме принятых решений

- **Продукт:** армибилдер GW на BSData; опоры — UI, чистая валидация, offline-first.
- **Стек:** TypeScript-монорепо (RN/Expo + Node + Next.js).
- **Движок:** Rust-парсер (лёгкий IR) + полноценный TS-движок вычисления (клиент+сервер) + app-слой оверрайдов (в т.ч. house-rules-режим).
- **Ростер — два состояния:** живой / иммутабельный submission-снапшот; развилка закладывается в Фазе 1.
- **Данные:** local-first + опциональная регистрация; шаринг без аккаунта; поля читаемости (имя/counts-as/заметки).
- **Контент:** data-driven, подписанный CDN + оверрайды (каталоги и скоринг).
- **Фазы:** 1) Билдер (+ Reference Mode в 1b) → **2a События/Проведение** → **2b Живой счёт** → 3) Полные турниры.
- **Имя:** коднейм **Astronomican Lighthouse** (внутр.); публичное имя TBD к релизу, GW-безопасное (кандидаты Muster/Levy).
- **Визуал:** дизайн-система задана — [тёмная](../../design/design-system-dark.md) + [светлая](../../design/design-system-light.md) (v1, §9.3); остаётся hi-fi макеты и верификация светлой на экранах.
- **Монетизация:** полностью бесплатно, без рекламы, без пейволлов.
- **Локализация:** i18n с первого дня, старт EN+RU.
- **Вычеркнуто:** нарративные/кампанийные форматы (Crusade/Path to Glory/KT spec ops); полный трекер коллекции/покраса (но лёгкий owned-флаг — кандидат позже); Play Mode/веб-страницы/HTML-шаблоны (заменены на счёт/турниры).
