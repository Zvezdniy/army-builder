# Группы choose-N (лимиты выбора) — Дизайн

**Дата:** 2026-07-11
**Статус:** Одобрено, готово к планированию
**Область:** Muster (engine-parser + engine-eval + domain)

## Цель

Смоделировать и проверять лимиты **choose-N** из BattleScribe-групп
`<selectionEntryGroup>` — по количеству выбранного. Это паттерн вооружения /
экипировки «выбери 1 из {Power Sword, Power Axe, Bolter}». Сейчас такие лимиты
теряются при разборе: участники группы «расплющиваются» в поле `children`
владеющего юнита, а собственный лимит группы отбрасывается с диагностикой
`group.constraint_dropped`. В итоге лимит выбора никогда не проверяется, и
нелегальная сборка (два оружия там, где разрешено одно) проходит как валидная.

## Рамки

**В рамках**
- Лимиты группы с `type` ∈ {`min`, `max`} по `field=selections`
  (выбрать не меньше / не больше N участников).
- Полноценное понятие «группа», сохранённое через IR до движка-валидатора.
- Точная, «громкая» обработка всего, что вне рамок (диагностика + drop, никогда
  не мапим молча, никогда не считаем неправильно).

**Явно вне рамок** (без изменений: диагностика-drop через
`group.constraint_dropped`)
- Лимиты группы по `field=points` (напр. «не больше 30 очков апгрейдов из группы»).
- Модификаторы на лимите группы (условные min/max).
- Вложенные под-группы как отдельные единицы choose-N. *Участники* под-групп
  по-прежнему «всплывают» в `children` владеющего юнита, как сейчас; собственные
  лимиты под-группы отбрасываются.

## Почему отдельный узел группы (решение)

Лимит choose-N агрегирует суммарное количество выбранного по **произвольному
набору соседних юнитов** (участникам группы) под одним владельцем. Это не
одиночный `entry` и не `category`, а его семантика «посчитай участников под этим
владельцем» не ложится на существующий enum областей `self/parent/force/roster`.
Отвергнутая альтернатива — перегрузить `IrConstraint` списком id участников плюс
особый разбор области видимости: это кладёт поле, нужное только группам, в общий
тип constraint и требует трансляции области, где легко посчитать неправильно.
Отдельный узел `IrGroup` оставляет общий `IrConstraint` чистым, даёт понятию
собственную честную семантику, сохраняет имя группы для внятных сообщений об
ошибках и естественно расширяется, если потом добавим очки / вложенные группы.

## Модель данных — `@muster/domain`

Новый минимальный тип лимита группы (без `scope`/`targetType`/`targetId`/
`modifiers` — набор участников и семантика «под этим владельцем» и так заданы
группой, поэтому эти поля были бы «мёртвым грузом» и источником ошибок):

```ts
export const IrGroupConstraint = z.object({
  id: z.string(),                       // сохраняем для overrides + идентичности ошибки
  type: z.enum(["min", "max"]),
  value: z.number().finite(),           // граница доверия к недоверенному входу
});
export type IrGroupConstraint = z.infer<typeof IrGroupConstraint>;

export const IrGroup = z.object({
  id: z.string(),
  name: z.string(),
  memberEntryIds: z.array(z.string()).default([]),
  constraints: z.array(IrGroupConstraint).default([]),
});
export type IrGroup = z.infer<typeof IrGroup>;
```

`IrEntry` получает одно поле:

```ts
groups: IrGroup[];   // z.array(IrGroup).default([]) в lazy-схеме; default [] в интерфейсе
```

Сторона Rust (`engine-parser/src/ir/model.rs`) зеркалит это serde-структурами
(`IrGroup`, `IrGroupConstraint`), с `#[serde(rename_all = "camelCase")]` и
`skip_serializing_if = "Vec::is_empty"` на `groups`/`memberEntryIds`/
`constraints` — чтобы юниты без групп сериализовались так же, как сейчас.

## Парсер — `engine-parser`

`map_entry` (в `src/ir/map.rs`) сейчас вызывает `collect_group_entries`, который
«расплющивает» участников в `children` и отбрасывает каждый лимит группы. Меняем:

- Участники по-прежнему «расплющиваются» в `children` (их очки и вложенные
  лимиты должны продолжать считаться) **и** id прямых участников каждой группы
  запоминаются.
- Для каждого `RawGroup` строим `IrGroup`, если у него есть **хотя бы один
  пригодный лимит min/max по selections**:
  - `id`, `name` из сырой группы.
  - `memberEntryIds` = id **прямых** участников группы, взятые из
    разрешённой (resolved) сырой модели (чтобы отражать инлайненные entryLinks).
    Потомки под-групп не включаются (вне рамок).
  - `constraints` = каждый сырой лимит группы с `type` ∈ {min, max} **и** чей
    `field` мапится в `selections`, как `IrGroupConstraint { id, type, value }`.
    `scope` сырого лимита **не** учитывается (семантика группы фиксирована).
- Любой лимит группы, который не мапится (field=points, неизвестный type, есть
  модификатор) → диагностика `group.constraint_dropped`, как сейчас. Группа без
  единого пригодного лимита **не** порождает `IrGroup` (поведение как сегодня —
  участники расплющены, лимит отброшен), так что пустых/«мёртвых» узлов групп
  не появляется.

При реализации проверить, что `resolve()` инлайнит entryLinks внутри групп; если
участник на момент маппинга — неразрешённый линк, его id всё равно нужно
захватить, чтобы движок мог его сматчить (добавить точечный тест).

## Валидатор — `engine-eval`

Отдельный проход, отдельно от цикла по `entry.constraints`, добавляется в
`evaluate()` после существующего цикла по лимитам юнитов:

```
for (const node of state.all)
  for (const group of node.entry.groups)
    for (const gc of group.constraints)
      issue = checkGroupConstraint(gc, node, group, state)
      if (issue) raw.push(issue)
```

`checkGroupConstraint` (без `costOf` — choose-N суммирует `selections`/количества,
никогда не очки) агрегирует по **прямым детям** узла-владельца, чей `entry.id` ∈
`group.memberEntryIds`, суммируя `effectiveCount`:

- `actual = Σ effectiveCount` подходящих прямых детей.
- `max` нарушен, когда `actual > value`; `min` нарушен, когда `actual < value`.
- Прямые дети — это правильно: участники группы «расплющены» как прямые дети
  владеющего юнита, а дерево ростера зеркалит дерево юнитов, поэтому выбор
  участника — прямой ребёнок выбора владельца.
- Каждое вхождение владеющего юнита (выбран несколько раз / под разными
  родителями) — это свой `EvalNode`, и он проверяется независимо.

Выдаваемый `Issue`:
- `code`: `group.max` / `group.min` (отдельно от `constraint.*`, чтобы UI мог
  показать нарушения групп по-особому).
- `message`: называет группу, напр. `Too many in "Wargear": 2 exceeds max 1` /
  `Not enough in "Wargear": 0 below min 1`.
- `constraintId`: `gc.id`; `entryId`: `node.entry.id` владельца; `selectionId`:
  `node.selectionId` владельца.

Overrides и модификаторы цены не трогаем: issue группы проходит по существующему
пути гашения через override (по ключу `constraintId` + опциональный
`selectionId`) ровно как issue лимита юнита/форса.

## Тестирование

- **domain:** round-trip Zod для `IrGroup`/`IrGroupConstraint`; дефолты
  (`groups` отсутствует → `[]`); отклонение не-finite `value`; отклонение
  неизвестного `type`.
- **parser (`tests/map.rs`):** группы только-min, только-max и min+max → корректный
  `IrGroup` с участниками и лимитами; лимит группы `field=points` →
  `group.constraint_dropped` + нет `IrGroup`; группа с модификатором →
  отброшена; участники по-прежнему присутствуют в `children`; лимит вложенной
  под-группы отброшен, при этом её участники всплывают.
- **engine (`test/`):** choose-max выполнен (1 → ok) и нарушен (2 → error);
  choose-min выполнен и нарушен (0 при min 1 → error); override гасит нарушение
  группы; группа, чьи участники не выбраны, не даёт ложного срабатывания на
  `max` (и даёт корректное нарушение `min`, когда min ≥ 1).
- **e2e / контракт:** обновлённый golden совпадает; ростер в контракте вызывает
  `group.max`.
- Сохранить принудительное **100%** покрытие statements/branches/functions/lines
  на обоих TS-пакетах; Rust-набор тестов держать зелёным.

## Глобальные ограничения (обязательны для каждой задачи)

- **Никогда не считать неправильно.** Всё, что не представимо точно, — громкая
  диагностика + drop, никогда не угаданное значение.
- **Недоверенный вход.** Все новые числовые IR-поля — `z.number().finite()`;
  парсер не добавляет `unsafe` (крейт остаётся `#![forbid(unsafe_code)]`).
- **Контракт синхронен.** Golden парсера и копия golden в engine-eval
  обновляются вместе; golden-тест и кросс-языковой контракт-тест оба проходят.
- **Покрытие.** Принудительные пороги 100% на `@muster/domain` и
  `@muster/engine-eval` остаются зелёными; `pnpm typecheck`, `cargo test`,
  `cargo deny check`, `cargo audit` — все чистые.
