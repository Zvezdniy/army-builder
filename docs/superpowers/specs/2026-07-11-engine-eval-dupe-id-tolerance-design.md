# Дизайн: терпимость engine-eval к дубликатам entry id (минимальный анблок реальных каталогов)

**Дата:** 2026-07-11
**Статус:** утверждён к реализации
**Связано:** [[real-catalogue-web-probe]], [[engine-parser-status]], `2026-07-11-real-catalogue-recon.md`

## Проблема

Резолв парсера **инлайнит** shared-сущности: `entryLink` → рекурсивная копия целевой `sharedSelectionEntry` в каждое место ссылки (`resolve/links.rs`, `resolve_entry` делает `entry.clone()` с **неизменным id**). На реальном каталоге Space Marines это даёт тысячи копий с одинаковыми GUID-ами по всему дереву.

`@muster/engine-eval` строит таблицу символов `Map<id, IrEntry>` (`symbols.ts::buildSymbolTable`) и на первом же дубликате id **бросает** `Duplicate entry id in catalogue: …`. Из-за этого `evaluate()` падает сразу — реальный каталог невозможно провалидировать/посчитать, а веб-билдер белеет при попытке добавить реальный юнит.

Подтверждено в Node: парс обрезанного реального IR + `evaluate()` на пустом ростере бросает немедленно на id `128d-c573-273f-47b5`.

## Границы (что делаем / чего НЕ делаем)

**Делаем — только один файл `packages/engine-eval/src/symbols.ts`:**
- `buildSymbolTable` перестаёт слепо бросать на дубликате id; вместо этого решает, штатный это инлайн или настоящая коллизия.

**НЕ делаем (отдельные срезы, отложено):**
- Парсер и форма IR не трогаются (никакого reference-IR / dedup-инлайна на стороне Rust).
- Структурное ужатие 30–63 MB — отдельный срез позже, если реально мешает.
- `categoryNames` / `ruleTexts` эмиссия — отдельный срез.
- Перф загрузки веба (skip-Zod / виртуализация) — отдельный срез.
- Диагностики conditions/modifiers (100k) — отдельный срез (корректность очков, не этот анблок).

## Поведение

`buildSymbolTable` обходит дерево `entry.children` рекурсивно, как сейчас. Изменяется только ветка «id уже в таблице»:

1. **id новый** → регистрируем, обходим детей (как сейчас).
2. **id уже есть, и сущности структурно равны** (штатный инлайн одной и той же shared-сущности) → **первая побеждает**: не перерегистрируем, **не обходим детей** этой копии (они идентичны и уже зарегистрированы под первым вхождением). Побочный эффект — обход становится O(уникальные), а не O(все инлайн-копии): попутный перф-выигрыш и отсутствие каскада «дубликат ребёнка».
3. **id уже есть, но сущности НЕ равны** (два разных определения под одним id — на реальных данных не должно случаться) → **по-прежнему бросаем** `Error` с этим id. Сохраняем принцип №1 «никогда не считать неправильно» — молчаливый first-wins на реально различающихся определениях недопустим.

### Почему first-wins безопасен на равных копиях

Копии одного GUID байт-идентичны: резолв — чистое клонирование (`entry.clone()`), а межфайловые id `merge_supporting` дедупит **до** резолва. Значит любые два вхождения одного id — это одна и та же сущность с одинаковым поддеревом. Регистрировать её один раз семантически эквивалентно текущему поведению на каталогах без дубликатов, но не падает на каталогах с инлайном.

### Структурное равенство

Рекурсивное сравнение двух `IrEntry` (id, имя, cost, constraints, groups, profiles, children — всё, что влияет на evaluate). Реализация — детерминированный deep-equal по полям `IrEntry`; можно опереться на сериализацию (`JSON.stringify` двух поддеревьев) как на простой и заведомо корректный компаратор, т.к. форма IR — обычный JSON без циклов и без порядок-зависимых Map/Set. Вызывается только на вхождениях-дубликатах; на равной копии мы дальше не рекурсируем, поэтому сравнение поддерева каждого дубликата делается один раз.

## Тесты (TDD, `symbols.test.ts` / при необходимости `evaluate.test.ts`)

1. **Штатный инлайн:** каталог, где одна shared-сущность (с детьми) присутствует дважды под одинаковым id и идентична → `buildSymbolTable` строится без ошибки, id в таблице ровно один раз, значение = первое вхождение; `evaluate()` на ростере отрабатывает.
2. **Настоящая коллизия:** две сущности с одним id, но разными полями (напр. разный cost) → `buildSymbolTable` **бросает** `Duplicate entry id in catalogue: <id>` (инвариант сохранён).
3. **Дедуп детей:** дубликат-родитель, чьи дети тоже были бы дубликатами → без ошибки (дети не переобходятся), таблица содержит детей один раз.
4. **Регресс:** существующие golden/contract/`mini40k` (без дубликатов) — зелёные, поведение не изменилось.

## Осязаемая проверка (не в CI, руками)

- Node: парс обрезанного реального SM IR + `evaluate()` на пустом ростере → без падения.
- Веб-билдер: загрузить реальный IR, добавить реальный юнит → нет белого экрана, datasheet рендерится, очки считаются.

Реальный GW-IP IR в git не коммитим (остаётся в scratchpad/gitignored `apps/web/public/`).

## Риски

- **Настоящая коллизия проскочит как «равная»** — исключено: при неравенстве бросаем.
- **Перф deep-equal** — сравнение только на дубликатах, поддерево каждого дубликата обходится один раз; для листовых Abilities (основная масса) сравнение дешёвое; крупных shared-групп мало. Разовая стоимость на построении таблицы, приемлемо.
- **assertDepth** — глубина считается по пути; пропуск детей равного дубликата держит глубину в тех же рамках.
