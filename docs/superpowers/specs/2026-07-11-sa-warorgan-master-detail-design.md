# S-A: War Organ-style master-detail layout — дизайн

**Дата:** 2026-07-11
**Фаза:** A дорожной карты War-Organ-parity (раскладка билдера). Утверждён пользователем «вариант A» на кликабельном макете.
**Цель:** перестроить веб-билдер в master-detail как у War Organ — слева ростер по ролям, справа полный datasheet выбранного юнита; на телефоне — drill-in. Плюс два улучшения, которые делают раскладку «настоящей»: авто-модели (юнит приходит с моделями → статлайн виден сразу) и имена ролей/keywords (categories id→name).

## Что видит пользователь (утверждено на макете)

- **Слева** — твой ростер, сгруппированный по ролям (HQ / Battleline / …), каждая строка: имя юнита + число моделей; снизу «+ добавить юнит». Клик по строке фокусирует юнит.
- **Справа** — datasheet выбранного: keywords-пилюли, статлайн-чипы, Wargear (choose-N), таблицы оружия, Abilities с описанием, кнопка remove.
- **Телефон**: та же раскладка складывается в drill-in — виден список; тап по юниту показывает его datasheet с кнопкой «назад».

## Архитектура

### 1. Домен: имена категорий (`packages/domain/src/ir.ts`)

`IrCatalogue` получает опциональное `categoryNames?: Record<string,string>` (id→имя), `z.record(z.string()).default({})`. Резолвит и **роли** (группировка ростера), и **keywords** (пилюли) в человекочитаемые имена. Опционально (default `{}`) — парсер сейчас его не эмитит (реальные каталоги получат имена, когда дойдём до их загрузки; вне объёма). Демо-каталог `mini40k.ir.json` заполняет map вручную (`cat.hq`→"HQ", `cat.troops`→"Battleline", `cat.model`→"Model").

### 2. `@muster/roster`: группировка + число моделей (`builder.ts`)

Чистые функции (логика в `builder.ts`, 100% покрытие):

```ts
export interface RoleGroup { role: string; units: RosterSelection[] }
export function unitsByRole(roster: Roster, catalogue: IrCatalogue): RoleGroup[]
export function modelCount(catalogue: IrCatalogue, selection: RosterSelection): number
```

- `unitsByRole` — группирует корневые `roster.selections` по **первой категории** их entry, резолвя имя через `catalogue.categoryNames` (fallback: сам id; если категорий нет — "Other"). Порядок ролей — по первому появлению; юниты внутри роли — в порядке добавления.
- `modelCount` — сумма по поддереву выбора `count` тех узлов, чей entry несёт профиль `typeName === "Unit"` (эвристика «модель = узел со статлайном»). Captain → 1; Battle Squad с Trooper×5 → 5.

Модель-как-Unit-профиль — та же эвристика, что и авто-модели (ниже). IR не несёт `type` (unit/model/upgrade), поэтому «модель» определяем по наличию Unit-статлайна.

### 3. `@muster/roster`: авто-модели (`initialChildren` в `builder.ts`)

При добавлении юнита он должен приходить с моделями (как в WO), чтобы статлайн был виден сразу. Расширяем `initialChildren(entry)`:

- вычисляем `unitMin` = значение constraint'а `field=selections, type=min, includeChildSelections=true` у entry (у squad-body это 5); если такого нет — правило не срабатывает;
- находим **модель-ребёнка** = первый `child`, чей entry несёт профиль `typeName==="Unit"` и который ещё не засеян существующими правилами;
- если `unitMin>=1` и модель-ребёнок найден — засеваем его `seedChild` с `count = unitMin`.

Существующее поведение (группы-дефолты, дети с own min≥1) сохраняется; авто-модель добавляется только когда её ещё нет. Captain (Unit-профиль на себе, без includeChildSelections-min) не затрагивается — его статлайн и так виден.

### 4. Демо-данные (`apps/web/src/mini40k.ir.json`)

- добавить `categoryNames` (см. п.1);
- дополнить **Battle Squad** и **Assault Squad** способностями (Abilities) и по одному weapon-профилю, чтобы их datasheet был содержательным (сейчас профили только у Captain/Trooper/wargear). Всё синтетика (GW-IP не тащим).

### 5. Веб: master-detail (`apps/web/src`)

- **`App.tsx`**: держит `selectedUnitId` (id корневого выбора в фокусе; по умолчанию — последний добавленный / первый в списке). Раскладка — CSS-грид: десктоп две колонки, телефон одна.
- **`RosterList.tsx`** (новый, левая колонка): рендерит `unitsByRole`; заголовки ролей; строки юнитов (имя + `modelCount` моделей), выбранная подсвечена; клик → `onSelect(id)`. Внизу — добавление юнита (переиспользуем `availableUnits`, простой список/кнопки «+ Name», по клику `addUnit` и фокус на новый).
- **`UnitDetail.tsx`** (новый, правая колонка): для выбранного юнита — keyword-пилюли (`entry.categories` через `categoryNames`), `Datasheet`, `UnitConfig` (настройка: группы/степперы/remove). На телефоне сверху кнопка «‹ назад» (сбрасывает выбор → показывает список).
- **`RosterPanel.tsx`**: остаётся источником `SelectionNode`/рендера вложенных опций (переиспользуется внутри `UnitDetail` для дерева выбора), или его роль поглощает `UnitDetail`. Точную границу решаем при реализации; важно не потерять существующие контролы (свопы/степперы) и очки/валидацию (общий счётчик + issues переезжает в шапку или левую колонку).
- **Точки/валидация**: общий `totalPoints/pointsLimit` и список issues остаются (в шапке над колонками). Пер-юнитные очки в строках — **вне объёма** (нет пер-юнитного расчёта в движке; показываем число моделей).
- **Респонсив**: `@media (max-width: 640px)` → одна колонка; когда `selectedUnitId` задан на узком экране, показываем деталь (с «назад»), иначе список.

## Тестирование

- **Домен**: `categoryNames` present/absent (default `{}`).
- **roster**: `unitsByRole` (группировка, порядок, fallback без категории/имени), `modelCount` (Captain=1, squad=5, дедуп не нужен — считаем count'ы), авто-модели (`addUnit` даёт squad с Trooper×min; Captain без авто-модели; без includeChildSelections-min не срабатывает). 100% покрытие.
- **web (jsdom)**: список группирует по ролям; клик по юниту меняет деталь; keyword-пилюли резолвятся в имена; добавление юнита появляется в нужной роли; (узкий экран — если проверяемо в jsdom, иначе визуально).
- Живая проверка в браузере: как у макета.

## Вне объёма (отложено)

- Пер-юнитные очки в строках (нужен пер-node расчёт в движке с учётом модификаторов).
- Эмит `categoryNames` парсером для реальных каталогов (сделаем при загрузке реальных данных).
- Primary-категория как «истинная роль» (сейчас берём первую категорию; IR не несёт `primary`-флаг — уточним при реальных данных).
- Detachment/enhancements (S2), attached units (WO «Leader»), экспорт/сохранение.
